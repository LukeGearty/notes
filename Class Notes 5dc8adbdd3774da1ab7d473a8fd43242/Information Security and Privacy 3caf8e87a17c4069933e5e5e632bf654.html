Web Browsers

  1. Web Browser as an OS: Browsers have evolved into platforms for a wide array of tasks, effectively functioning as an "operating system" for many online applications that previously were managed by local programs (e.g., email and document editing).
  1. Browser vs. OS Architecture: While browsers differ from traditional operating systems, several analogies exist:
    • System Calls (OS) vs. DOM (Browser): Browsers use the DOM (Document Object Model) to facilitate user requests, similar to how an OS uses system calls.
    • Process Isolation: Operating systems isolate processes for stability and security; browsers perform similar isolation between frames or tabs.
    • Storage Management: Operating systems rely on disk-based storage, while browsers use mechanisms like cookies or local storage.
    • Access Control: Operating systems typically mediate access by user IDs, while browsers control access by content origins.
  1. Chrome’s Security Architecture: Chrome’s architecture includes security features designed to prevent entire browser crashes and improve stability:
    • Browser Kernel: Functions like an OS kernel with full privileges, handling the browser’s core operations.
    • Rendering Engines: These components handle page content layout and are isolated in separate processes, helping contain and mitigate vulnerabilities. Since most vulnerabilities occur within the rendering engine, process isolation enhances security by preventing crashes in one tab from affecting the entire browser.
  1. Advantages of Chrome’s Isolation:
    • Chrome’s isolation model prevents complete browser crashes when one tab or plugin fails, displaying an “Aw Snap” message for the affected tab only.
    • Compared to Firefox at the time, which would crash all open windows and tabs on error, Chrome’s architecture enables faster recovery for individual tabs and a more stable experience.

Information Flow Control:

Key Points on Access Control & Information Flow Control (IFC)

  1. Traditional Access Control Approaches:
    • Access Control Lists (ACLs): Restrict access by checking if a person is on a pre-defined list. Only those authenticated individuals on this list can access sensitive information, like a credit card number.
    • Capability-Based Systems: Access is granted based on possession of a specific capability or ticket, which acts as proof of access rights.
  1. Information Flow Control (IFC):
    • Concept: Instead of controlling who can access information, IFC allows anyone to access data but restricts their ability to share or communicate it. For example, IFC would let anyone see a credit card number but then monitor them to ensure they don’t disclose it.
    • Challenges: Preventing data leaks goes beyond stopping direct access and requires preventing indirect or gradual disclosures of information.
  1. Covert Channels & Timing Channels:
    • Covert Channels: Methods used by processes to secretly convey information, typically by using indirect signals outside the normal information flow (e.g., timing of resource usage).
    • Timing Channels: Information is leaked based on the timing of actions, such as the speed or order of responses, which can reveal sensitive details.
  1. Example of a Covert Channel:
    • By saying the name “Justin” with subtle differences, a person can hint at hidden information, like which hand a remote is in, without directly disclosing it.
    • Similarly, a process can use timing or resource patterns to leak information covertly, making it difficult to secure systems against unintended data disclosure.
  1. Practical Challenges of IFC:
    • Real-world systems struggle to eliminate covert and timing channels entirely, making complete information control nearly impossible.
    • Example with Passwords: Systems must convey whether a login was successful or not. This inherently reveals some information about the password, which can’t be entirely hidden.
    • Declassification Requirement: Special programs are needed to manage and “declassify” certain information, making it accessible under controlled circumstances. These programs must be carefully designed to avoid accidental leaks.

Conclusion

While IFC is an intriguing concept, it has significant practical limitations due to the inherent difficulty in preventing covert and timing channels. The reliance on specially designed, secure declassification programs makes IFC complex and potentially unreliable for practical security applications.

1. Remote Attestation:
◦
Definition: A process that allows a device to prove to a remote party that it’s running untampered software.
◦
Mechanism: The Trusted Platform Module (TPM) takes snapshots of system memory during the boot process, generates cryptographic hashes, and signs them with a hardware-embedded private key.
◦
Trust: If the remote party trusts the manufacturer and the TPM’s attestation, they can believe that the system is running the specified software version.
2.
Limitations of TPM:
◦
Software Vulnerabilities: TPM can only verify the version of software running, but it doesn’t protect against vulnerabilities within that software that attackers could exploit.
◦
Physical Attack Risks: TPM doesn’t prevent attacks from physical keyloggers, USB devices, or other hardware tampering.
◦
Attestation Limitations: While remote attestation adds trust, it’s not foolproof in situations like using unknown computers (e.g., internet cafes) for sensitive tasks.
3.
Physical Security of TPM:
◦
Tamper Detection: To protect the private key, TPMs often include mechanisms to self-destruct or erase sensitive data if tampering is detected.
◦
Challenges: Securing TPMs against well-funded, sophisticated adversaries is difficult, as they may use advanced techniques (e.g., chip surface shaving) to access the data.

Lecture: Access Control in Virtual Machines and Virtualization

1. Introduction to Virtualization and Access Control:

  • Focus: Access control within virtual machines (VMs) and virtualization.
  • Key question: How does an operating system (OS) manage access control and isolate resources?

2. Evolution of Operating Systems:

  • Early OS examples:
    • Commodore 64, early MS-DOS (no graphical interface, hard drive, or mouse).
    • Only one program could run at a time; no multitasking or isolation features.
    • If a program froze or entered an infinite loop, the entire system required a restart.
  • Modern OS advancements:
    • Support for running multiple processes simultaneously.
    • Built-in process isolation and resource management.
    • Provision of a programming abstraction (e.g., system call interface in Linux, Win32 DLL in Windows).

3. Key Functions of an OS:

  • Process Isolation: Ensures processes don’t interfere with each other.
  • Programming Abstraction: Provides a standard interface for tasks like opening files, reading, and writing data (e.g., system calls in Linux).
    • Challenge: Broad interfaces with hundreds of system calls are complex to secure and restrict.
  • Resource Restriction: Limits resource consumption to prevent buggy or malicious processes from damaging the system.

4. Virtualization and Access Control Challenges:

  • Reasoning about Permissions:
    • Virtualization simplifies reasoning about permissions by separating systems with unique OS kernels.
    • Access control in shared systems (e.g., single OS managing multiple users) is harder to secure.
  • Programming Abstractions in VMs:
    • Virtualization offers isolated abstractions within separate virtual environments, simplifying resource management.

Lecture: Resource Isolation and Denial of Service (DoS) Attacks

1. Introduction to Resource Isolation:

  • Goal: Allow multiple processes to run on a system without interfering significantly with each other.
  • Importance: Prevents one process from monopolizing system resources, which could otherwise lead to denial of service (DoS).

2. Distributed Denial of Service (DDoS):

  • Classic DDoS: Overwhelms a website with excessive requests.
  • Modern Example (Mirai Botnet): Utilized IoT devices to flood popular sites with high-bandwidth traffic, resulting in more severe impact than past DDoS attacks.

3. Fork Bombs:

  • Definition: A script that spawns copies of itself indefinitely, leading to resource exhaustion.
  • Challenges:
    • Process Overload: Creates a race condition where killing processes is difficult as new ones spawn faster.
    • Login Process Limits: Prevents new processes from being created, making it hard to regain system control.
  • Mitigation: Reserving specific process resources for root users to prevent untrusted users from monopolizing all process IDs.

4. Memory Exhaustion Attack (APT Package Manager Vulnerability):

  • Vulnerability: APT would store excessive data in memory when downloading packages, potentially consuming all system memory.
  • Impact: Server slows, may swap data to disk, and eventually crashes due to memory depletion.
  • Root Cause: Lack of memory limit checks in the package manager.

5. Disk Exhaustion Attack (YUM Package Manager Vulnerability):

  • Vulnerability: YUM would store excessive data on disk, potentially filling up the entire disk.
  • Impact: System crashes and behaves erratically due to lack of disk space, affecting system functions even after a reboot.

6. HTTP Range Request Attack (Apache Web Server):

  • Vulnerability: Apache would load a file into memory for each range request, even if only small portions were requested.
  • Attack: Multiple small range requests could cause the server to load numerous copies of a large file, overloading memory.
  • Mitigation: Limit file loading to one copy and restrict repeated large file requests.

7. SSL Handshake Renegotiation Attack:

  • Vulnerability: SSL/TLS protocol allowed clients to request key renegotiation repeatedly, consuming server resources.
  • Impact: Server spends excessive time on repeated cryptographic operations, leading to performance degradation.
  • Mitigation: Limit renegotiation requests to reduce server workload.

8. Mitigation Strategies in Resource Isolation:

  • Process isolation, such as Chrome’s separation of rendering processes, helps fine-tune controls and restrict resource usage to specific processes, providing better DoS resistance and enhanced system stability.

Summary:

Resource isolation is essential for modern operating systems to manage resources effectively and prevent various types of DoS attacks. Vulnerabilities in system processes and protocols (e.g., fork bombs, memory exhaustion, SSL renegotiation) can lead to resource depletion, but targeted mitigations and process separation can help mitigate these risks.

Lecture: Types of Operating System Virtual Machines (VMs)

Types of OS VMs:

  1. Type 1A Virtualization:
    • Structure: Hardware → Hypervisor → Multiple OS instances (full OS kernels) → Applications.
    • Characteristics: Each OS instance appears to have its own dedicated hardware (e.g., 1 CPU, 1 GB of RAM).
    • Pros: Easier to allocate and manage resources, clearer security and performance boundaries.
    • Cons: High overhead due to running multiple OS kernels, limits scalability.
  1. Type 1B Virtualization:
    • Structure: Hardware with a single, shared OS kernel that uses Virtual Machine IDs (VMIDs) for isolation.
    • Characteristics: All virtual instances share the same OS, with extra data in kernel structures (e.g., processes have VMIDs).
    • Pros: Lightweight, supports high scalability (hundreds to thousands of VMs).
    • Cons: Challenging to achieve strong security and isolation, especially for performance.
  1. Type 2 Virtualization:
    • Structure: Host OS → Hypervisor application (e.g., VMware Workstation, VirtualBox) → Guest OS with applications.
    • Characteristics: Hypervisor operates as an application on an existing OS, running guest OSes within it.
    • Pros: Can run on unmodified systems; flexible and convenient for end-users.
    • Cons: High overhead; complex interfaces between the host OS, hypervisor, and guest OS, which complicates security and performance management.

Containerization:

  • Technology Examples: Docker, containerD.
  • Characteristics: Runs applications within isolated containers, each with a customized environment (often on top of Linux-based Type 1B virtualization).
  • Pros: Low overhead, broad compatibility across devices, encapsulates application dependencies.
  • Cons: Limited performance isolation, complex interfaces due to shared OS elements, similar security concerns as Type 1B and Type 2 VMs.

Summary:

  • Advantages of OS Virtualization:
    • Compatibility with legacy applications.
    • Simplified security reasoning in OS VMs vs. directly on OS.
    • Cross-OS compatibility for certain applications (e.g., running Windows apps on Linux).
  • Limitations:
    • Processor architecture dependency.
    • High overhead in Type 1A and Type 2 due to multiple OS kernels.
    • Type 1B, Type 2, and containers face challenges in performance and security isolation due to complex OS interfaces.

Containerization - picture shipping containers

Containerization is a way to package up software and everything it needs to run, like code, libraries, and dependencies, into a single "container."

A container is a lightweight, isolated environment that can run on any system, regardless of that system's underlying setup. Because containers include all the necessary components for the software to run, you don't have to worry about compatibility issues—whether it's on your laptop, in a data center, or on a cloud server, the containerized app works the same way.

In short, containers help make software portable and consistent across different environments, simplifying deployment and scaling.

Taking it a step further, containerization is a way of virtualizing operating systems to make applications run consistently across different environments. Containers are often compared to virtual machines (VMs), but they’re much more lightweight. While VMs virtualize the hardware (running a full operating system for each app), containers virtualize the operating system itself, sharing the OS kernel among containers. This makes containers faster to start up and use fewer resources than VMs.

Here's how it works:

  1. Isolation: Each container runs as an isolated process on the host OS, with its own file system, network, and libraries. This allows containers to avoid interfering with each other.
  1. Portability: Because each container bundles the application along with its dependencies, it can be run consistently across different environments (development, testing, production) without modification.
  1. Efficiency: Containers share the host OS kernel and use only what they need, meaning they consume less memory and CPU than a VM would.

Containers help development teams avoid the "works on my machine" problem and make deploying, scaling, and updating applications faster and easier. Popular tools like Docker make managing and deploying containers simple, while orchestrators like Kubernetes help manage many containers at scale.

Think of it like this: instead of needing several physical computers, you can use virtualization software to create multiple virtual computers on one machine. Virtualization is a technology that allows a single physical machine (like a server or computer) to host multiple virtual machines (VMs), each running its own operating system and applications independently. This is done using a software layer called a hypervisor that sits between the physical hardware and the virtual machines. The hypervisor allocates resources (like CPU, memory, and storage) to each VM, making them function as if they were on separate physical machines.

Here’s how it works and why it’s useful:

  1. Resource Optimization: Without virtualization, a lot of a computer’s resources can go underused. Virtualization allows multiple VMs to share the same hardware, making better use of resources and saving money.
  1. Isolation: Each VM is isolated from others, so a crash or error in one VM doesn’t affect the others. This is important for security and reliability.
  1. Flexibility and Scalability: Virtual machines can be created, duplicated, and moved quickly, allowing businesses to scale up computing power or adapt to changing needs.
  1. Multi-Environment Testing: Developers can use VMs to test software across different operating systems without needing separate physical hardware for each one.

Type 1 Hypervisor (Bare-Metal)

  • Definition: A Type 1 hypervisor, or "bare-metal" hypervisor, runs directly on the physical hardware of a host machine. It doesn’t require a host operating system, making it more efficient because it has direct access to the hardware.
  • Examples: VMware ESXi, Microsoft Hyper-V, and Xen.
  • Use Cases: Often used in data centers and enterprise environments for high-performance, large-scale virtualization needs. Type 1 hypervisors are known for being fast, secure, and stable since they don’t rely on a host OS.

Type 2 Hypervisor (Hosted)

  • Definition: A Type 2 hypervisor, or "hosted" hypervisor, runs on top of an existing operating system. The host OS manages the hardware, and the Type 2 hypervisor runs as an application on this OS, creating and managing VMs.
  • Examples: VMware Workstation, Oracle VirtualBox, and Parallels Desktop.
  • Use Cases: Primarily used for smaller-scale or individual use, like testing and development on personal computers. Type 2 hypervisors are easier to set up and work well on devices where you want to run multiple operating systems for testing or development.

Key Differences

  • Performance: Type 1 hypervisors generally offer better performance since they interact directly with the hardware, bypassing a host OS.
  • Use: Type 1 is preferred in professional, high-demand environments, while Type 2 is more common for personal or small-scale setups.
  • Overhead: Type 2 hypervisors have more overhead since they rely on a host OS to manage hardware, which can make them slower than Type 1 in certain cases.

  • Cross-Site Scripting (XSS): A common web attack where an attacker injects malicious scripts (usually JavaScript) into webpages viewed by other users.
  • Prevalence: XSS is widespread on platforms that allow user-generated content (e.g., social media, blogs, forums) as they often display user input to other users.

  • Vulnerable Server-Side Code: The attack typically targets a web server with inadequate input sanitization.
  • Execution on Other Users' Browsers: Malicious scripts posted by one user are executed by other users’ browsers as though they came from the trusted website.

Software updates are important, but when the server used to give updates is compromised, that is a huge problem. It is critical to protect those repositories

  1. Importance of Updates: Software updates are crucial to fix vulnerabilities that attackers could exploit remotely. However, a commonly overlooked risk is when the repository hosting these updates is compromised, allowing attackers to push malware to many users at once.
  1. Repository Compromise Risks: Compromised repositories can have massive impacts, as seen with incidents like the NotPetya ransomware attack, which used updates to spread malware globally, impacting even multinational corporations and posing risks to cyber-physical systems like self-driving cars.

    • Many repositories use "online keys" (keys stored on the repository) to sign updates during transmission. While transport mechanisms like SSL/TLS can protect against man-in-the-middle attacks, they don’t secure updates if the repository itself is breached.

  1. Offline Keys as an Alternative:
    • Some repositories use offline keys (stored off-repository, e.g., encrypted on a USB stick in a safe deposit box) to add security by preventing attackers from signing updates if they breach the repository.
    • Offline keys use cryptographic systems like GPG or RSA, ensuring that even if servers are compromised, tampering is detectable.
  1. Persistent Risks and Strong Threat Model:
    • Major tech companies and free software organizations have faced attacks via software updates. If they are vulnerable, it indicates that securing software updates is challenging and a compromise is likely inevitable.
    • TUF operates on a strong threat model assuming eventual compromise and aims to limit the impact even if attackers access the repository and signing keys.

Design principles of TUF

Different types of metadata assigned by different roles using different keys so that a compromised key doesn’t affect the entire system. “Don’t put all your eggs in 1 basket”.

  1. Foundation of TUF:
    • TUF (The Update Framework) is designed to limit the impact of a repository compromise and ensure recovery.
    • Principles include separation of duties, threshold key signing, key revocation, offline keys, and cryptographic diversity.
  1. Design Principles:
    • Separation of Duties:
      • Different types of metadata are signed by different roles with unique keys to limit the impact of key compromise.
      • TUF uses four top-level roles, managed by administrators, with distinct functions:
        • Targets Role: Manages metadata about packages, including hashes and file sizes. Administrators may delegate signing responsibility to developers or other roles.
        • Snapshot Role: Provides a global version of the repository, tracking different metadata file versions for consistency.
        • Timestamp Role: Summarizes if metadata or packages have been updated with a cryptographic hash of the snapshot file.
        • Root Role: Distributes keys for all top-level roles and functions as the root certificate authority (like CA in SSL/TLS).
      • Delegations allow administrators to manage keys, providing a way to distribute, revoke, or replace keys.
    • Threshold Key Signing:
      • For critical actions (e.g., signing for important binaries), a minimum number of keys are required to sign a file.
      • This approach ensures that multiple keys are needed to launch a high-impact action, like launching a nuclear missile analogy.
    • Key Revocation:
      • TUF allows for explicit key revocation by publishing new keys to replace old ones.
      • Alternatively, key expiration can be set, requiring administrators to renew keys to avoid expired keys compromising security.
    • Offline Keys:
      • High-value roles (e.g., root, targets) should use offline keys to minimize risk, keeping keys off the repository in secure storage (e.g., USB sticks in a safe deposit box).
    • Diversity of Cryptographic Algorithms:
      • TUF supports multiple hashing and signing algorithms to hedge against vulnerabilities in a single algorithm.
      • For example, using multiple algorithms (e.g., SHA-2 and SHA-3) provides security even if one is compromised.

Buffer Overflow Attack Overview

  • Basic Concept:
    • A buffer overflow occurs when an input exceeds the allocated memory space, potentially overwriting crucial parts of the program, such as instructions or return addresses.
    • This vulnerability allows an attacker to inject code or redirect program execution to unintended functions, potentially bypassing security mechanisms (e.g., bypassing password checks).

Mechanics of Buffer Overflow

  1. Program and Stack Layout:
    • Programs have specific memory allocations for user inputs (buffers) and return addresses.
    • Example: A user input buffer might be 1KB. If the user enters more than 1KB, it can overflow, overwriting other parts of the stack.
  1. Control Flow Hijacking:
    • By overwriting the return address on the stack, attackers can redirect execution to malicious code or other code segments they are not supposed to access.
    • Initial attack methods often involved placing the malicious code in the overflowed buffer and redirecting the return to execute it.

Mitigation Techniques

  1. Non-Executable Stack (NX Bit/DEP):
    • The stack memory is marked as non-executable to prevent code execution from buffer overflowed areas.
    • Limits some attack vectors but doesn’t completely prevent attackers from exploiting the system.
  1. Safe Input Handling:
    • Using functions with bounded input sizes to prevent excessive input from overflowing the buffer.
  1. Canary Values:
    • Special values (canaries) placed before return addresses; if overwritten, the program detects it and aborts.
    • Effective but can be bypassed if attackers have methods to read or manipulate canaries.
  1. Static Analysis Tools:
    • Tools identify potential buffer overflow vulnerabilities in code during development, allowing for early detection and patching.
  1. Address Space Layout Randomization (ASLR):
    • Randomizes memory layout, making it harder for attackers to predict memory addresses.
    • A common and lightweight mitigation that reduces the success of overflow attacks by obscuring memory structure.

XSS - Cross Site Scripting

Cross-site scripting (also known as XSS) is a web security vulnerability that allows an attacker to compromise the interactions that users have with a vulnerable application. It allows an attacker to circumvent the same origin policy, which is designed to segregate different websites from each other. Cross-site scripting vulnerabilities normally allow an attacker to masquerade as a victim user, to carry out any actions that the user is able to perform, and to access any of the user's data. If the victim user has privileged access within the application, then the attacker might be able to gain full control over all of the application's functionality and data.

Cross-Site Scripting (XSS) is a security vulnerability that allows an attacker to inject malicious scripts into a legitimate website or web application, causing it to execute code in the browser of another user who visits the affected page. These scripts can manipulate the webpage, steal data (like session cookies, which coulrd lead to account hijacking), or redirect users to malicious sites.

Here's how it works in detail:

  1. Injection: The attacker identifies a part of the web application where user input isn't properly sanitized or validated, like a comment box or URL parameter.
  1. Payload Delivery: The attacker injects a malicious script (e.g., JavaScript) into this input field. When the input isn't sanitized, the script gets stored (in stored XSS) or reflected back (in reflected XSS) to any visitor who views that part of the site.
  1. Execution: When a user loads the affected page, the script executes in their browser, with the same permissions as the legitimate site, leading to potential data theft, session hijacking, or even malware distribution.

Types of XSS:

  • Stored XSS: The payload is saved on the server (e.g., in a database) and executed whenever a user loads the page.
  • Reflected XSS: The payload is sent as part of the request (e.g., URL parameter) and immediately reflected back to the user by the server.
  • DOM-based XSS: The vulnerability occurs directly within the browser, often through client-side JavaScript without any server interaction.

Preventing XSS requires robust input validation and output encoding, ensuring that any user-provided content is treated as text rather than executable code. Proper Content Security Policies (CSPs) can also limit the impact of XSS attacks.

Prevention

1. Stored XSS

In a blog website with a comments section, suppose the site doesn't sanitize user input properly. An attacker might post a comment like:

html
Copy code
<script>alert("Hacked!");</script>

This script tag gets stored in the website's database. Now, whenever someone visits the blog post and loads the comments, the script runs in their browser, displaying the "Hacked!" alert. Since it’s saved on the server, anyone who visits that page will trigger the script.

2. Reflected XSS

In this case, the website immediately reflects user input back to the page without sanitization. Imagine a search function where the search term shows up on the results page, like:

bash
Copy code
example.com/search?query=hello

The results page might display: “You searched for: hello.” An attacker could trick someone into clicking a link like this:

javascript
Copy code
example.com/search?query=<script>alert("Hacked!");</script>

When the page loads, the script is reflected back and executed in the victim's browser, triggering the alert. This only affects users who click on the specially crafted link.

3. DOM-based XSS

DOM-based XSS occurs entirely within the browser, usually due to JavaScript code manipulating the page’s Document Object Model (DOM) with untrusted data. Imagine a website with a page that takes a #user fragment in the URL and displays it on the page:

document.getElementById("welcome").innerHTML = "Welcome, " + location.hash.substring(1);

If someone visits example.com#welcome<script>alert('Hacked!')</script>, the JavaScript inserts the <script> tag directly into the DOM. The browser then executes the alert script, resulting in XSS.

Summary:

  • Stored XSS: Attacker’s script is saved on the server and executes whenever a user loads the page.
  • Reflected XSS: Attacker’s script is reflected back immediately based on the URL or request parameters.
  • DOM-based XSS: The attack happens directly in the user’s browser through client-side JavaScript that modifies the page’s DOM.

Prevention Tips:

  • Stored and Reflected XSS are often mitigated with server-side sanitization and output encoding.
  • DOM-based XSS is handled by properly validating and sanitizing inputs in JavaScript and avoiding direct insertion of untrusted data into the DOM.

Anonymity -

Overview of Tor (The Onion Router):

  • Tor is an anonymity network, an open-source software used by millions for anonymous browsing.
  • It prevents websites from identifying users, ensuring privacy.
  • Developed collaboratively with contributions from various entities, including government bodies (U.S.) and non-profits like the EFF and ACLU.

Purpose of Tor:

  • Used to protect users from surveillance by anonymizing their online activity.
  • Popular among those wanting privacy from government surveillance, corporations, advertisers, and hackers.

The Threat Model:

  • In Tor's context, the "attacker" is often a powerful entity (e.g., governments, ISPs, corporations) trying to monitor users.
  • Attackers may attempt to:
    • Observe who is communicating (such as “Alice” and “Bob”).
    • Impersonate or take over websites.
    • Control parts of the Tor network to monitor traffic.

Limitations of Cryptography Alone:

  • While encryption protects the contents of messages, it doesn’t conceal the identity of the parties communicating.
  • ISPs or network observers can see where traffic is going (e.g., specific websites) even if they can’t read the data.
  • Anonymity requires more than just encryption; it needs a secure routing mechanism like Tor provides.

Privacy Concerns and Surveillance:

  • Regular users are often subject to tracking by advertisers and companies seeking data for targeted marketing.
  • Governments and advertising companies have lax data protection, leading to potential privacy risks.
  • Example: Germany exposed sensitive data about public officials who supported advertising data collection, highlighting risks in data leaks.

Benefits of a Large Anonymity Network:

  • A larger Tor network increases anonymity strength. With many relay nodes, it becomes harder to track users.
  • A single anonymous server is weak (easily traceable), whereas a large, distributed network complicates tracing.

How Tor Achieves Anonymity:

  • Tor anonymizes traffic by routing it through multiple nodes (relays), effectively “hopping” across the network.
  • This process, known as “onion routing,” hides the origin and destination, enhancing user privacy.

Notes on Bitcoin and Security

Security in Large-Scale Distributed Systems

  • Challenge:
    • Defining a formal security notion for large-scale distributed systems is difficult and often unfeasible.
    • Security is approached through threat modeling:
      • Divide the system into modules.
      • Secure each module and its interactions.
      • Provide a partial guarantee of overall system security.
  • Bitcoin's Security Approach:
    • Security is often equated to stability:
      • The system must continue functioning despite motivated attackers (e.g., due to financial incentives).
    • Bitcoin’s security is analyzed across three components:
      1. Transactions
      1. Blockchain
      1. Peer-to-Peer Network

Security of Transactions

  • Transaction Validation Rules:
    • Miners validate transactions before adding them to the blockchain.
    • Requirements include:
      • Correct transaction format as per specifications.
      • Valid signature created with the private key controlling the input.
      • Valid destination addresses.
    • Stability:
      • Validation rules must remain consistent so all miners agree on valid transactions.

Security of the Blockchain

  • Consensus Protocol:
    • Heavily studied in literature, with several formally defined security properties:
      1. Eventual Consensus:
        • Miners may initially disagree on recent blockchain history.
        • Over time, they converge to the longest chain.
      1. Exponential Convergence:
        • Probability of an attacker rewriting the blockchain decreases exponentially over time.
        • Longer chains require significantly more computational power to modify.
      1. Growth (Liveness):
        • The blockchain must grow with new blocks added regularly.
        • Frozen blockchain indicates a denial-of-service (DoS) attack.
      1. Correctness:
        • Blockchain must include only valid blocks and transactions.
        • Invalid blocks or transactions compromise security.
      1. Fairness:
        • Mining rewards should correspond to mining power.
        • Security issues arise with techniques like block withholding, which can unfairly increase rewards.

Security of the Peer-to-Peer Network

  • Availability:
    • Network reliability ensures miners receive updates on transactions and blocks.
    • Issues in network reliability can lead to:
      • Miners working on different blockchain branches, violating eventual consensus.
      • Frozen blockchains due to dropped blocks or transactions.
    • A robust network is critical for blockchain security and stability.

Conclusion

  • Bitcoin’s security depends on the stability and functionality of its three core components: transactions, blockchain, and the peer-to-peer network.
  • Ensuring security for each module contributes to the overall stability of the system.
