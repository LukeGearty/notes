The OSI (Open Systems Interconnection) model is a conceptual framework that describes how data is transmitted across a network. It divides the process into seven layers, each with specific roles. Here’s an overview with simple explanations for each layer:

1. Physical Layer

What it does: Handles the physical connection between devices and the transmission of raw binary data (0s and 1s) over cables, radio waves, or other mediums.

Examples: Ethernet cables, fiber optics, Wi-Fi signals.

Simple Explanation: Think of it as the wires or wireless signals that carry data from one device to another.

2. Data Link Layer

What it does: Ensures error-free data transfer between devices on the same network. It organizes data into frames and manages access to the physical medium.

Examples: MAC addresses, switches.

Simple Explanation: Like labeling envelopes so data gets to the correct device within the local network.

3. Network Layer

What it does: Determines the best path for data to travel from one device to another, even across multiple networks. It handles logical addressing.

Examples: IP addresses, routers.

Simple Explanation: Like GPS for data, figuring out the best route to its destination.

4. Transport Layer

What it does: Ensures reliable data delivery between devices by managing error checking, retransmissions, and flow control.

Examples: TCP (Transmission Control Protocol), UDP (User Datagram Protocol).

Simple Explanation: Like a delivery service, ensuring the package arrives safely and in order.

5. Session Layer

What it does: Manages and controls the connections between devices, keeping communication sessions open, secure, and synchronized.

Examples: Session management in APIs, login systems.

Simple Explanation: Like a phone call, maintaining the connection until both sides hang up.

6. Presentation Layer

What it does: Translates data into a format the application layer can understand. It also handles encryption and compression.

Examples: JPEG, MP3, SSL/TLS encryption.

Simple Explanation: Like a translator, ensuring the data is in a readable format for the application.

7. Application Layer

What it does: Provides the interface for end-users and applications to interact with the network.

Examples: Web browsers (HTTP), email clients (SMTP), file transfer programs (FTP).

Simple Explanation: Like the app you use to browse the web, send emails, or watch videos.

Mnemonic for the Layers:

To help remember the order, you can use a phrase like:

“Please Do Not Throw Sausage Pizza Away” (Physical, Data Link, Network, Transport, Session, Presentation, Application).

Step 1: Application Layer (Layer 7)

The user writes an email using an application like Gmail or Outlook.

The email is prepared and follows protocols like SMTP (Simple Mail Transfer Protocol).

What happens here: The application prepares the email for sending.

Step 2: Presentation Layer (Layer 6)

The email data is formatted (e.g., plain text, HTML, or attachments) and possibly encrypted (e.g., TLS/SSL).

What happens here: The email is converted into a readable and secure format for transmission.

Step 3: Session Layer (Layer 5)

A session is established between the sender’s and receiver’s devices to maintain communication.

What happens here: The system manages the connection so the email can flow smoothly.

Step 4: Transport Layer (Layer 4)

The email is broken into smaller pieces called segments. Protocols like TCP ensure reliable delivery by numbering these segments and requesting retransmission if any get lost.

What happens here: Data is prepared for transport with reliability checks.

Step 5: Network Layer (Layer 3)

Each segment is assigned a source and destination IP address (e.g., 192.168.1.1) and packaged into packets.

Routers decide the best path for the packets to take across networks to reach the recipient’s email server.

What happens here: The packets navigate through the internet toward the destination.

Step 6: Data Link Layer (Layer 2)

The packets are encapsulated into frames with MAC addresses for the devices on the same local network. Frames ensure proper delivery within the local or immediate network.

What happens here: Frames carry the data over the local network, such as through a switch or Wi-Fi access point.

Step 7: Physical Layer (Layer 1)

The frames are converted into electrical signals, light pulses (fiber optics), or radio waves (Wi-Fi) and transmitted through cables, antennas, or other physical media.

What happens here: Raw bits are sent over the medium (e.g., Ethernet cables or Wi-Fi).

At the Receiving End

1. The recipient’s device follows the reverse process:

Physical Layer: Receives raw bits via cables or Wi-Fi.

Data Link Layer: Converts bits into frames and passes them up.

Network Layer: Uses the destination IP address to route packets to the correct device.

Transport Layer: Reassembles the segments into the original data.

Session Layer: Maintains the communication session.

Presentation Layer: Decodes, decompresses, and decrypts the email.

Application Layer: Displays the email to the user in their email application.

Client-Server Architecture

Client-server architecture is a way computers communicate with each other in a network.

• The client is the device or program (like your web browser) that requests a service or resource.

• The server is the device or program that provides the requested service or resource (like a website or file).

Example:

1. You (the client) type a website URL in your browser.

2. The server hosting the website sends the data back to your browser.

3. You see the website load on your screen.

In short: Clients ask, servers respond

In Peer-to-Peer (P2P) architecture, devices (called peers) communicate directly with each other, without needing a central server to manage requests. Each peer can act as both a client and a server, sharing resources like files, processing power, or data.

Example:

1. You use a file-sharing program to download a song.

2. The file is split into pieces, and each piece is downloaded from different peers who have the file.

3. At the same time, your computer can share pieces of files it already has with others.

In short: Peers share directly with each other, without a middleman.

Overview

  • HTTP: A protocol used for transferring hypertext (web pages) between a client (e.g., browser) and a server.
  • Stateless: Each request/response is independent, meaning the server does not retain session information unless explicitly implemented using cookies, sessions, or other mechanisms.
  • Operates on the application layer of the OSI model.

HTTP Methods

  1. GET: Requests data from a server.
    • Used to retrieve resources (e.g., HTML, JSON, images).
    • Parameters can be sent in the URL as query strings.
    • Safe and idempotent (does not modify data).
  1. POST: Submits data to a server to process.
    • Often used for form submissions.
    • Data is sent in the body of the request.
    • Not idempotent (can result in changes or side effects).
  1. PUT: Updates a resource on the server.
    • If the resource does not exist, it may create one.
    • Idempotent.
  1. DELETE: Deletes a resource from the server.
    • Idempotent.
  1. HEAD: Similar to GET but only retrieves headers, not the body.
    • Useful for checking if a resource is available.
  1. OPTIONS: Describes communication options for a resource.
    • Used in CORS (Cross-Origin Resource Sharing) preflight requests.
  1. PATCH: Partially updates a resource.
    • Non-idempotent.

HTTP Request Structure

  1. Request Line:
    • Format: <Method> <URL> <HTTP Version>
    • Example: GET /index.html HTTP/1.1
  1. Headers:
    • Key-value pairs providing metadata about the request.
    • Examples:
      • User-Agent: Information about the client making the request.
      • Accept: Types of content the client can process (e.g., text/html, application/json).
      • Authorization: Credentials for authentication.
  1. Body (Optional):
    • Contains data sent to the server (e.g., form data in POST requests).

HTTP Response Structure

  1. Status Line:
    • Format: <HTTP Version> <Status Code> <Reason Phrase>
    • Example: HTTP/1.1 200 OK
  1. Headers:
    • Provide metadata about the response.
    • Examples:
      • Content-Type: Format of the response (e.g., text/html).
      • Set-Cookie: Set cookies for the client.
  1. Body (Optional):
    • Contains the data returned by the server (e.g., HTML, JSON).

HTTP Status Codes

  • 1xx (Informational): Request received, continuing process.
    • 100 Continue
  • 2xx (Success): Request processed successfully.
    • 200 OK: Standard response for successful requests.
    • 201 Created: Resource successfully created.
  • 3xx (Redirection): Further action needed to complete the request.
    • 301 Moved Permanently - Requested object has been permanently moved; the new URL is specified in Location: header of the response message. The client software will automatically retrieve the URl
    • 302 Found
    • 304 Not Modified
  • 4xx (Client Errors): Problem with the request.
    • 400 Bad Request
    • 401 Unauthorized: Authentication needed.
    • 403 Forbidden: Access denied.
    • 404 Not Found: Resource not found.
  • 5xx (Server Errors): Problem with the server.
    • 500 Internal Server Error
    • 502 Bad Gateway
    • 503 Service Unavailable

HTTP Versions

  1. HTTP/1.0:
    • One request per connection (no connection reuse).
    • Limited header options.
  1. HTTP/1.1:
    • Persistent connections (keep-alive).
    • Chunked transfer encoding.
    • Support for caching and virtual hosting.
  1. HTTP/2:
    • Binary protocol (not text-based like HTTP/1.x).
    • Multiplexing: Multiple requests and responses over a single connection.
    • Header compression for better performance.
  1. HTTP/3:
    • Based on QUIC protocol.
    • Reduces latency and improves performance over unreliable networks.

Security in HTTP

  • HTTPS: Secure version of HTTP using TLS/SSL for encryption.
    • Encrypts data in transit to prevent eavesdropping and tampering.
    • Identified by the https:// prefix and a padlock icon in browsers.
  • Common Headers:
    • Strict-Transport-Security (HSTS): Enforces HTTPS.
    • Content-Security-Policy (CSP): Prevents injection attacks.
    • X-Frame-Options: Prevents clickjacking.

Key Concepts

  • Cookies: Small pieces of data stored by the browser for state management (e.g., sessions).
  • Caching: Mechanisms like ETag and Cache-Control to reduce server load and improve speed.
  • RESTful APIs: HTTP used as a foundation for APIs adhering to REST principles.

HTTPS

Hyper Text Transfer Protocol Secure

Combination of HTTP and SSL (Secure Socket Layer) protocols.

Widely used.

Overview

  • HTTPS is the secure version of HTTP, adding encryption, data integrity, and authentication using TLS/SSL.
  • Ensures secure communication between a client (e.g., browser) and a server.
  • Identified by the https:// protocol in URLs and a padlock icon in browsers.

Core Features

  1. Encryption:
    • Uses TLS (Transport Layer Security) or its predecessor SSL (Secure Sockets Layer).
    • Ensures data transmitted between client and server is encrypted and not readable by attackers.
  1. Authentication:
    • Uses digital certificates to confirm the server's identity.
    • Certificates are issued by trusted Certificate Authorities (CAs).
  1. Data Integrity:
    • Prevents data from being tampered with during transmission.
    • Ensures the data received is identical to the data sent.

TLS/SSL Handshake

  • A process that establishes a secure connection between client and server.
  1. Client Hello:
    • Client sends supported cryptographic algorithms and a random value to the server.
  1. Server Hello:
    • Server selects algorithms, provides its certificate, and sends a random value.
  1. Certificate Validation:
    • Client validates the server’s certificate using CA-signed certificates.
  1. Key Exchange:
    • Server and client exchange keys securely (e.g., using RSA or Diffie-Hellman).
  1. Session Key Established:
    • Both agree on a symmetric session key for encrypting further communication.
  1. Secure Connection:
    • Encrypted data exchange begins.

Digital Certificates

  • X.509 Certificates:
    • Standard format for HTTPS certificates.
    • Contains the server’s public key, domain name, expiration date, and the issuing CA's information.
  • Certificate Authority (CA):
    • Trusted entities that issue and validate certificates.
    • Examples: DigiCert, Let’s Encrypt, GlobalSign.
  • Certificate Chain:
    • Includes intermediate and root certificates to verify the legitimacy of the server’s certificate.

Key HTTPS Headers

  • Strict-Transport-Security (HSTS):
    • Forces clients to use HTTPS, even if HTTP is initially requested.
    • Example: Strict-Transport-Security: max-age=31536000; includeSubDomains.
  • Public-Key-Pinning (HPKP):
    • Binds a website to specific cryptographic keys to prevent impersonation (deprecated due to complexity).
  • Content-Security-Policy (CSP):
    • Mitigates cross-site scripting (XSS) and other code injection attacks.
    • Example: Content-Security-Policy: default-src 'self';.

Advantages of HTTPS

  1. Security:
    • Encrypts all data, including sensitive information (e.g., passwords, credit card numbers).
  1. Trust:
    • Increases user confidence through visible indicators (e.g., padlock icon).
  1. SEO:
    • Search engines like Google prioritize HTTPS-enabled websites.
  1. Performance:
    • Modern HTTPS implementations (e.g., HTTP/2 with TLS) often outperform HTTP.
  1. Compliance:
    • Meets security standards for regulations like GDPR, PCI DSS, and HIPAA.

How to Enable HTTPS

  1. Obtain a Certificate:
    • Get an SSL/TLS certificate from a CA.
    • Free options are available (e.g., Let’s Encrypt).
  1. Configure the Server:
    • Install the certificate on the web server.
    • Update server configuration to support HTTPS.
  1. Redirect HTTP to HTTPS:
    • Use .htaccess rules or server settings to force HTTPS.
  1. Test Configuration:
    • Use tools like SSL Labs or OpenSSL to validate the setup.

Common HTTPS Issues

  1. Mixed Content:
    • Occurs when a secure HTTPS page includes resources (e.g., images, scripts) loaded over HTTP.
    • Fix: Update all resources to use HTTPS.
  1. Certificate Errors:
    • Expired certificates, mismatched domain names, or untrusted CAs.
    • Fix: Regularly renew certificates and ensure proper configurations.
  1. Performance Overhead:
    • HTTPS adds slight latency due to encryption/decryption.
    • Mitigation: Use optimized TLS libraries and HTTP/2.

TLS/SSL Vulnerabilities

  1. Old Protocols:
    • SSL 2.0, SSL 3.0, and TLS 1.0/1.1 are outdated and insecure.
    • Fix: Use TLS 1.2 or TLS 1.3.
  1. Man-in-the-Middle (MITM) Attacks:
    • Attacks targeting unencrypted or poorly implemented HTTPS.
    • Fix: Use HSTS and strong ciphers.
  1. Weak Cipher Suites:
    • Vulnerabilities in encryption algorithms (e.g., RC4, DES).
    • Fix: Disable weak ciphers in server configuration.

Modern TLS Features

  • TLS 1.3:
    • Faster handshake and reduced latency.
    • Enhanced security with modern ciphers and no support for older algorithms.
  • Perfect Forward Secrecy (PFS):
    • Protects past communications even if the server’s private key is compromised.

FTP

  1. Purpose: FTP is used to transfer files between a client (your computer) and a server (another computer).
  1. Connection:
    • Your computer connects to an FTP server using an FTP client (a program like FileZilla).
    • You log in with a username and password or use anonymous access if the server allows it.
  1. Commands:
    • The client sends commands to the server to do things like list files, upload, or download files.
    • The server responds and performs the requested actions.
  1. Two Modes of Communication:
    • Control Channel: This is for sending commands and responses (e.g., "list files").
    • Data Channel: This is for transferring the actual files.
  1. Two Transfer Modes:
    • Active Mode: The server connects back to the client to send data.
    • Passive Mode: The client connects to the server for both commands and data. (This is more common for modern setups.)
  1. File Transfer:
    • You can upload files (send them to the server) or download files (get them from the server).
  1. Security Note: Basic FTP doesn’t encrypt data, so anyone can intercept the data. For secure transfers, use FTPS (FTP Secure) or SFTP (SSH File Transfer Protocol).

FTP uses two main ports:

  1. Port 21: This is the control port. It handles the commands and responses between the client and server, like logging in, listing files, or starting a transfer.
  1. Port 20: This is the data port. It is used for transferring the actual files in active mode.

Email Overview

  • Email is a method of sending messages electronically from one user to another using the Internet.
  • It involves sending, receiving, and storing messages on mail servers.

Email Protocols

  1. SMTP (Simple Mail Transfer Protocol):
    • Purpose: Used for sending emails from the client to the server or between servers.
    • Port Numbers:
      • Port 25: Default (but often blocked to prevent spam).
      • Port 587: For sending emails securely (STARTTLS encryption).
      • Port 465: Legacy port for sending with SSL encryption.
    • Flow: Client → Mail Server → Recipient's Mail Server.
  1. POP3 (Post Office Protocol version 3):
    • Purpose: Used to retrieve emails from the server to the client.
    • Characteristics:
      • Downloads emails to the client and often deletes them from the server.
      • Best for offline access, but not suitable for accessing emails from multiple devices.
    • Port Numbers:
      • Port 110: Default.
      • Port 995: Secure connection with SSL/TLS.
    • Flow: Server → Client (download).
  1. IMAP (Internet Message Access Protocol):
    • Purpose: Used to retrieve and manage emails on the server.
    • Characteristics:
      • Synchronizes emails between client and server, allowing access from multiple devices.
      • Emails remain on the server unless explicitly deleted.
      • Better for modern, multi-device workflows.
    • Port Numbers:
      • Port 143: Default.
      • Port 993: Secure connection with SSL/TLS.
    • Flow: Server ↔ Client (synchronization).

Web Mail

  • Purpose: Access email through a web browser.
  • Examples: Gmail, Outlook.com, Yahoo Mail.
  • Emails are stored on the provider's servers and accessed through a web interface.
  • Advantages:
    • No need for a separate client application.
    • Accessible from anywhere with an internet connection.
  • Disadvantages:
    • Reliant on the provider's availability.
    • Requires internet access to read and manage emails (unless cached locally).

DNS

What is DNS?

  • DNS (Domain Name System) is the system that translates human-readable domain names (e.g., www.example.com) into IP addresses (e.g., 192.168.1.1) so that browsers and other applications can locate resources on the Internet.
  • DNS is essential for routing traffic on the Internet, as users typically access websites using domain names rather than IP addresses.

How DNS Works

  1. DNS Query Process:
    • When a user types a domain name (e.g., www.example.com) into their browser, the browser sends a DNS query to a DNS resolver.
    • The DNS resolver is usually provided by the ISP or a third-party service (e.g., Google DNS or Cloudflare).
    • The resolver checks its cache for the IP address associated with the domain.
      • If the IP is not found in the cache, it queries other DNS servers to resolve the address.
      • The resolver will query root DNS servers, TLD (Top-Level Domain) servers, and finally the authoritative DNS servers for the domain to get the IP address.
  1. DNS Resolution Steps:
    • Step 1: The DNS resolver checks if it has the requested domain's IP address cached.
    • Step 2: If not cached, the resolver queries a root DNS server, which directs it to the appropriate TLD server (e.g., .com for example.com).
    • Step 3: The TLD server directs the query to the authoritative DNS server for the domain, which provides the IP address for the domain.
    • Step 4: The resolver sends the IP address back to the client (e.g., the web browser), which then connects to the server hosting the website.

Types of DNS Records

  1. A Record (Address Record):
    • Maps a domain name to an IPv4 address (e.g., example.com → 192.168.1.1).
  1. AAAA Record (IPv6 Address Record):
    • Maps a domain name to an IPv6 address (e.g., example.com → 2001:0db8:85a3:0000:0000:8a2e:0370:7334).
  1. CNAME Record (Canonical Name Record):
    • Alias record that points one domain to another domain (e.g., www.example.com → example.com).
  1. MX Record (Mail Exchange Record):
    • Defines the mail servers responsible for receiving emails for the domain (e.g., example.com mail handled by mail.example.com).
  1. NS Record (Name Server Record):
    • Indicates which DNS servers are authoritative for the domain.
  1. PTR Record (Pointer Record):
    • Used for reverse DNS lookups, mapping an IP address to a domain name (used in security and troubleshooting).
  1. TXT Record:
    • Can contain arbitrary text, commonly used for verification, security (e.g., SPF, DKIM, DMARC), or other purposes.

DNS Caching

  • Caching is an optimization that stores DNS query results locally for a certain period (TTL, or Time to Live) to reduce query traffic and speed up name resolution.
  • TTL specifies how long a record is cached before a new query is made to the authoritative server.
  • Both the client and the resolver cache DNS queries, and the records remain cached until the TTL expires.

DNS Security

  1. DNS Spoofing (Cache Poisoning):
    • Attackers insert false DNS records into a cache, causing clients to resolve incorrect IP addresses.
    • Mitigated by DNSSEC (DNS Security Extensions) which adds cryptographic signatures to DNS records to ensure their authenticity.
  1. DNSSEC (DNS Security Extensions):
    • A set of extensions to DNS that adds security by using digital signatures to verify the integrity of the DNS data and prevent tampering.
    • Ensures that DNS records are authentic and have not been altered in transit.
  1. DDoS Attacks on DNS:
    • DNS Amplification Attacks: Attackers use DNS servers to flood a target with massive amounts of traffic, overwhelming its resources.

DNS Servers

  1. Recursive DNS Server:
    • Responsible for resolving domain names by querying other DNS servers.
    • It typically belongs to an ISP or third-party DNS service.
  1. Authoritative DNS Server:
    • Contains the official DNS records for a domain. When queried, it provides the correct and most up-to-date information for that domain.
  1. Root DNS Servers:
    • The top-level servers in the DNS hierarchy, responsible for directing queries to appropriate TLD servers.
  1. TLD DNS Servers:
    • Manage the domain extensions (e.g., .com, .org, .net) and point to authoritative servers for domains under that extension.

DNS Tools

  • nslookup: A command-line tool to query DNS servers and troubleshoot DNS issues.
  • dig: A more advanced tool for querying DNS records and gathering information about DNS servers.
  • whois: A tool to query domain registration information, which includes nameservers.

Common DNS Issues

  1. DNS Resolution Failures: Occur when the DNS server cannot find the IP address for a domain.
  1. Slow DNS Resolution: Caused by issues like an overloaded DNS server, poor network connection, or misconfigured settings.
  1. DNS Propagation Delay: When DNS records are updated, changes may take time (up to 48 hours) to propagate across all servers globally.

Transmission Control Protocol (TCP)

  • Purpose: Reliable communication for transmitting data between devices on a network.
  • Features:
    • Connection-Oriented: Establishes a connection before transmitting data (via a three-way handshake).
    • Reliable Delivery: Ensures data reaches its destination correctly.
      • Includes error checking and retransmission for lost packets.
    • Ordered Delivery: Reassembles data in the correct order at the receiver.
    • Flow Control: Manages the rate of data transmission between sender and receiver.
    • Examples: Used in applications like HTTP (web browsing), FTP (file transfer), and email (SMTP).

User Datagram Protocol (UDP)

  • Purpose: Fast, lightweight communication with minimal overhead.
  • Features:
    • Connectionless: No need to establish a connection before sending data.
    • Unreliable Delivery: No guarantee that data will arrive, nor in what order.
    • Low Overhead: Faster than TCP due to the lack of reliability features.
    • Applications: Ideal for real-time applications like video streaming, gaming, and VoIP (Voice over IP).
    • Examples: DNS, DHCP, TFTP.

Internet Protocol (IP)

  • Purpose: Handles addressing and routing of packets across networks.
  • Features:
    • Addressing: Assigns unique IP addresses to devices for identification.
      • IPv4: 32-bit addresses (e.g., 192.168.1.1).
      • IPv6: 128-bit addresses (e.g., 2001:0db8:85a3:0000:0000:8a2e:0370:7334).
    • Routing: Directs packets from source to destination using routers.
    • Connectionless: Operates independently of transport protocols like TCP or UDP.
    • Fragmentation: Splits large packets into smaller ones to fit the network’s maximum transmission unit (MTU).
    • Protocol Suite: Works with protocols like TCP and UDP to provide end-to-end communication.

RFC for TCP

Three Way Handshake

A process used by TCP to establish a reliable connection

Steps of the Three-Way Handshake

  1. SYN (Synchronize)
    • The client initiates the connection by sending a SYN packet to the server.
    • This packet includes the initial sequence number (ISN), which the client will use to track data transmission.

    Purpose: Request to establish a connection.

  1. SYN-ACK (Synchronize-Acknowledge)
    • The server responds to the client’s SYN with a SYN-ACK packet.
    • This packet acknowledges the client’s SYN request and provides the server’s ISN for the client to track the server’s data.

    Purpose: Acknowledge the connection request and send synchronization details.

  1. ACK (Acknowledge)
    • The client sends an ACK packet back to the server, acknowledging receipt of the server’s SYN-ACK.
    • After this step, the connection is established, and data transfer can begin.

    Purpose: Confirm the connection setup.

Illustration

  1. Client → Server: SYN (Seq = X)
  1. Server → Client: SYN-ACK (Seq = Y, Ack = X+1)
  1. Client → Server: ACK (Seq = X+1, Ack = Y+1)

Key Features

  • Sequence Numbers: Used to track the order of packets and ensure reliable delivery.
  • Acknowledgments: Indicate successful receipt of data.
  • Reliable Connection: Guarantees both parties agree to start communication with specific parameters.

Importance of the Three-Way Handshake

  1. Establishes a Reliable Connection: Ensures both the client and server are ready to communicate.
  1. Synchronization: Helps both parties agree on initial sequence numbers.
  1. Prevents Connection Hijacking: Confirms the identity and readiness of both the sender and receiver.

Common Issues

  • SYN Flood Attacks: A form of DoS (Denial of Service) attack where an attacker sends a large number of SYN requests without completing the handshake, overwhelming the server.
  • Timeouts: If one party doesn’t respond, the connection attempt may fail due to a timeout.

Subnetting

ubnetting is a fundamental networking concept used to divide a larger network into smaller, more manageable subnetworks. It helps optimize the use of IP addresses, enhance network performance, and improve security. Let’s break it down step-by-step.

1. IP Address Basics

  • An IP address is a 32-bit number divided into four octets (e.g., 192.168.1.1).
  • Each octet is separated by dots and can range from 0 to 255.
  • IP addresses have two parts:
    • Network part: Identifies the network.
    • Host part: Identifies a device within the network.

2. Subnet Masks

  • A subnet mask defines how the IP address is split into the network and host parts.
  • Example: 255.255.255.0 (CIDR notation: /24).
    • Each 255 in the subnet mask represents bits used for the network part.
    • Remaining bits (0s) represent the host part.

3. Why Subnet?

  • Efficient IP usage: Prevents wasting IP addresses.
  • Improved performance: Reduces traffic within a subnet.
  • Enhanced security: Segments networks for better control.

4. Calculating Subnets

Example Scenario:

Network: 192.168.1.0/24

You want to create 4 subnets.

Step 1: Determine the number of subnets

  • Formula: 2n, where n is the number of bits borrowed from the host portion.

    2n2^n

    nn

  • Borrow enough bits to accommodate your required subnets.
    • 22=42^2 = 422=4 (2 bits needed for 4 subnets).

Step 2: Update the subnet mask

  • Original mask: /24 (255.255.255.0).
  • Borrow 2 bits for subnets: /26 (255.255.255.192).

Step 3: Calculate subnet ranges

  • Each subnet increment is based on the block size: 2(32−subnet mask).

    2(32−subnet mask)2^{(32 - \text{subnet mask})}

  • Block size for /26: 26=64 IPs per subnet.

    26=642^{6} = 64

Subnets:

  1. 192.168.1.0 - 192.168.1.63
  1. 192.168.1.64 - 192.168.1.127
  1. 192.168.1.128 - 192.168.1.191
  1. 192.168.1.192 - 192.168.1.255

5. Key Terms

  • Subnet ID: The first IP in a range (e.g., 192.168.1.0).
  • Broadcast Address: The last IP in a range (e.g., 192.168.1.63).
  • Usable IPs: Exclude the subnet ID and broadcast address. For /26, there are 64−2=62 usable IPs.

    64−2=6264 - 2 = 62
