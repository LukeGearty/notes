Congestion Control

Congestion is when the network is overloaded

Router queues are full, so packets are dropped

or length of queues leads to long nodal queuing delay, causing timer to expire

Dropped packets lead to inefficiency and can compound the problem:

Congestion → Packet Loss → Retransmission → More congestion → More Loss!

Goal is to prevent the self-destructive feedback cycle above.

Better to wait than to send a packet likely to be dropped before reaching its destination.

We have end-to-end observations of network performance, but the precise internal cause of a network problem is difficult to know.

State of routers along path is unknown - end hosts know only their state.

Without Congestion Control

Problems arsie even in simple network with just one connection/flow

Shared Congestion causes more inneficiency

Observing Congestion

Recall that flow control prevents loss at receiver

If packets are dropped (timer expire before ACK), two possibilities:

Timer interval is too small (EWMA will adjust itself) or

The network is congested, and packets are being dropped by routers

Basic idea fo TCP congestion Control:

Sender tries to estimate an appropriate congestion window

Inf flight packets ≤ MIN(cwnd, rwnd)

Larger cwnd → greater throughput

When sender observes packet loss, decrease cwnd

TCP Reno

Very popular in 90s and 2000s, still in use today - not likely to be seen in Wireshark traces at homelabs

Allowed internet to scale by solving internet problems

Starts off with a slow congestion, and then each RTT the rate doubles. (Slow Start). Grows exponentially here.

Eventually the maximum is met, and we have an estimate of the maximum rate.

Speed is reduced, not to the beginning, cuts the sending rate in half. Additive Increase, Multiplicative Decrease.

Rate increases by one each RTT. Linear increase.

Eventually the maximum is met again, and then is halved again.

Rinse and repeat.

  1. Slow Start: initially, the network capacity is unknown, so cwnd = MSS (1460 bytes) send 1 packet
    1. Double cwnd each time an ACK is received (every RTT)
    1. This leads to exponential growth
    1. Loss by timeout → ssthresh = cwnd / 2
    1. if cwnd > ssthresh, move to cngestion avoidance
  1. In congestion Avoidance mode:
    1. ACK received → Cwnd += MSS
      1. Linear growth
    1. Loss by timeout → reset cwnd = MSS and move back to slow start mode
      1. Timeout indicates a serious network problem → go back to slow start
    1. On loss by triple-duplicate ACK: move to third phase:
  1. Fast Recovery Mode:
    1. Some packets are still getting through, so don’t over react. Cut cwnd in half

Incentive to follow TCP rules:

If I don’t throttle myself during congestion, then I will have poor performance in cases when I am the only host causing the congestion.

Hosts cannot observe the network directly, just observe packet losses.

Thus, everyone will reduce throughput when congestion occurs, just in case they are solely responsible for the congestion.

Also, there are only a few TCP implementations, supplied by OSes, so they generally follow an RFC’s recommendations.

Distance Vector

Each network node has its own DV.

For that node, DV lists the cost, next_hop for every destination.

Each node operates independently (a distributed algorithm)

Each node must store:

Its outbound links

Each neighbors latest DV

Each node must compute:

its own DV using 1 & 2. This will be optimal.

Whenever either 1 or 2 changes, I must recompute my DV

Whenever my DV changes, send my updated DV to all neighbors.

When link costs change, node detects local link cost change. Recalculates distance vector, DV notifies neighbors.

Good news travel quickly:

When link changes in a decrease, it is quickly updated for all neighbors.B

Bad news travels slowly, when a link costs increased, updates can be slow.

Neighbors still think that the other still has a good path and keep trying to route through each other, meanwhile each nde’s estimated distance slowly increases.

Multiple Access Protocols

Collision is the fundamental problem in broadcast links:

Multiple nodes try to communicate at the same time

Simultaneous messages interfere with each other

None of the colliding messages can be received

Precious time and bandwidth is wasted

Messages must be retransmitted

If we’re too aggressive in sharing, many collisions will occur

If we’re too polite about sharing, may waste time/bandwidth waiting.

Assuming multiple nodes are sharing a link with throughput R bps:

If only one node is communicating, then it should get the fulll bandwidth (R)

When N nodes have data to send, they should each get R/N bandwidth

Protocol should be decentralized with no single point of failure

It should be simple and inexpensive to implement
