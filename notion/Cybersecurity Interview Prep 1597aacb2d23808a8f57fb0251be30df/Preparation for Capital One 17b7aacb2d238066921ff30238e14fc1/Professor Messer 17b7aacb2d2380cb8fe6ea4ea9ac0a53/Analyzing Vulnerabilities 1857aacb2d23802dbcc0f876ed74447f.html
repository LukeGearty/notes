1. False Positives & False Negatives

  • False Positive:
    • A reported vulnerability that does not actually exist in the system.
    • Example: A vulnerability is flagged on an OS, but investigation shows it’s incorrect.
  • False Negative:
    • A real vulnerability exists but is not detected by the scanner.
    • More dangerous than false positives as attackers can exploit unnoticed vulnerabilities.

2. Reducing False Positives and False Negatives

  • Always update scanner signatures before performing a scan.
  • Contact the scanner manufacturer for clarification if results seem unclear.

3. Vulnerability Severity Categorization

  • Vulnerabilities are categorized into severity levels:
    • Critical, High, Medium, Low, or Informational.
    • Low/Informational vulnerabilities are still valid and not considered false positives.

4. CVE & CVSS for Prioritization

  • Common Vulnerabilities and Exposures (CVE):
    • A list of publicly disclosed vulnerabilities.
    • Example resources:
      • National Vulnerability Database (NVD).
      • CVE Database.
  • Common Vulnerability Scoring System (CVSS):
    • Scores vulnerabilities from 0 to 10 based on severity.
    • Versions: CVSS 2.0, CVSS 3.x, etc.

5. Vulnerability Scanners

  • Scan for vulnerabilities in:
    • Applications (desktop, mobile apps).
      • Example: CVE-2020-1889 (WhatsApp desktop bypass vulnerability).
    • Web Applications (hosted on servers).
      • Example: CVE-2020-24981 (UCMS access control vulnerability).
    • Network Devices (e.g., firewalls, routers).
      • Example: CVE-2020-2579 (D-Link software vulnerability).
  • Ensure scanners use the latest signatures to maximize accuracy.

6. Exposure Factor & Risk Assessment

  • Exposure Factor:
    • Represents risk as a percentage.
    • Example:
      • A vulnerability causing service downtime 50% of the time = 50% exposure.
      • A critical, unpatched public-facing vulnerability = 100% exposure.
  • Risk Tolerance:
    • Prioritize patching based on the organization’s willingness to accept risk.
    • Higher priority: Public-facing, revenue-generating, or easily exploitable systems.

7. Environmental Considerations for Patching

  • Factors influencing patch prioritization:
    • Public vs. Internal Systems: Public systems take precedence.
    • Critical Applications: Revenue-generating or business-critical apps are top priority.
    • Organization Type:
      • Example 1: A hospital might prioritize differently than a power generator.
      • Example 2: Ransomware can cause significant operational disruption (e.g., hospital closures or DDoS attacks on utilities).

8. Challenges of Patch Management

  • Patching cannot be done all at once; prioritize based on severity and risk.
  • Test patches thoroughly to ensure compatibility with the environment.
    • Testing vs. urgency:
      • For highly critical vulnerabilities, testing may need to be expedited.
