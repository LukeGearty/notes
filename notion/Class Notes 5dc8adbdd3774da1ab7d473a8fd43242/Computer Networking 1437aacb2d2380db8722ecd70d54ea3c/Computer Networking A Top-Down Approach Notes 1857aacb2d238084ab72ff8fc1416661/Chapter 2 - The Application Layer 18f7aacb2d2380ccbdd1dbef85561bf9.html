Chapter 2 - The Application Layer

Network Applications are the reason why a computer network exists - they are the reason why network infrastructure is built, and why protocols are created to support them.

Principles of Network Applications

At the core of network application development: writing programs that run on different end systems and communicate with each other over the network.

Software needs to be written on multiple systems - but you do not need to write software that runs on routers or link-layer switches.

Network Application Architectures

The network architecture - the internet protocol stack (Application, Transport, Network, Link, Physical) - is fixed. Application architecture is designed by the programmers and dictates how the application is structured over the various end systems.

Client-Server Architecture:

An always-on host (server) services requests from other hosts (clients)

A single-server host is sometimes incapable of keeping up with all the requests from clients (For example: YouTube, google, Facebook). A Data Center houses a large number of hosts - often used to create a powerful virtual server.

Google, Bing, Baidu, AMazon, eBay, Instagram, etc all run in Data Centers.

Peer to Peer Architecture:

Minimal to no reliance on dedicated servers in data centers. Instead, the application exploits direct communication between pairs of intermittently connected hosts - called peers.

The peers are controlled and owned by users -desktops and laptops - most peers residing in homes, universities, and offices.

The most compelling feature of P2P architecture is their self-scalability.

Example: BitTorrent

Processes Communicating

Process - an instance of a running program - a program running within an end system.

When processes run on the same end system, they can communicate with each other through the Interprocess Communication using rules governed by the end system’s operating system.

Processes running on different end systems communicate with each other by exchanging messages across the computer network.

A network application consists of pairs of processes that send messages to each other over a network.

For example, in the Web Application, a client browser process exchanges messages with a Web server process.

In a P2P file-sharing system, a file is transferred from a process in one peer to a process in another peer

For each pair of communicating processes, we label one of the two processes as the client and the other process as the server.

Web - browser is the client process, Web server is a server process

In the context of a communication session between a pair of processes, the process that initiates the communication is labeled as the client. The process that waits to be connected to begin the session is the server.

Any message that is sent from one process to another must go through the underyling network - a process sends messages into and receives messages from the network through a software interface called a socket.

As an analogy - a process is analogous to the house, and a socket is the house’s door

When a process wants to send a message to another process on another host, it shoves the message out its door (socket). The sending process assumes that there is a transportation infrastructure on the other side of its door that will transport the message to the door of the destination process. Once the message arrives at the destination host, the message passes through the receiving process’s door (socket) and the receiving process then acts on the message.

The socket is the API between the application adn the network, since the socket is the programming interface with which network applications are built.

Address Processes -

On the internet, the host is identified by its IP address - a 32-bit quantity that uniquely identifies the host

The sending process must also identify the receiving process and the receiving socket running in the host. This is where the port number is identified.

Transport Services Available to Applications

A socket is the interface between the application process and the transport-layer protocol. There are more than one transport-layer protocols available. Many networks provide more than one transport-layer protocols.

There are different services a transport-layer protocol can offer to applications invoking it.

Reliable Data Transfer

A packet can get lost within a computer network - packets can overflow a buffer in a router, or can be discarded by a host or router after having some of its bits corrupted.

Some applications cannot have this data loss - Email, financial applications, Web document transfers.

To support these applications, something has to be done to guarantee that the data sent by one end of the application is delivered correctly and completely to the other end of the application.

If a protocol guarantees data delivery, it is said to offer reliable data transfer. A sending process can pass data into its socket and guarantee that the data will arrive at the receiving process.

Loss-tolerant applications don’t need reliable data transfer, so some data may never arrive - Streaming, VoIP, etc. This is useful when lost packets won’t bring everything crashing down.

Throughput

Throughput is the rate at which the sending process can deliver bits to the receiving process.

The amount of data per second that can be transferred

Because other sessions will be sharing the bandwidth along the network path, and because these other sessions will be coming and going, the available throughput can fluctuate with time.

Because of this, another service a transport-layer protocol could provide is guaranteed available throughput at some specified rate.

The application can request a guaranteed throughput of r bits/sec.

Applications that have throughput requirements are bandwidth-sensitive applications

Elastic applications can make use of as little throughput or as much throughput as happens to be available.

Email, file transfer, web transfer.

“One cannot be too rich, too thin, or have too much throughput”.

Timing

Some services require sensitive timing - like internet telephony, or virtual conferencing or multiplayer games.

Security

Transport protocols can provide one or more security services - confidentiality (encryption), integrity and end-point authentication.

Transport Services Provided by the Internet

The types of transport services provided by the internet are TCP and UDP.

TCP

Connection-oriented - TCP has the client and server exchange transport-layer control information with each other before the application-level messages begin to flow. This is the TCP handshake/three-way handshake. After the handshake, a TCP connection is said to exist between the sockets of the two processes.

Reliable Data Transfer - the communicating processes can rely on TCP to deliver all data sent without error and in the proper order.

There is congestion control mechanism - it can throttle a sending process when the network is congested between sender and receiver.

To provide security for TCP, TLS was developed to provide security services.

UDP

UDP is lightweight transport protocol - it provides minimal services and does not guarantee delivery. There is no handshake. It is unreliable data transfer.

Services Not Provided By Internet Transport Protocols

Today’s internet can provide satisfactory service to time-sensitive applications, but it cannot provide any timing or throughput guarantees.

TCP can be easily enhanced at the application layer with TLS to provide security.

Application Layer Protocols

Application layer protocols define how an application’s processes, running on different end systems, pass messages to each other.

These define:

The types of messages exchanged

The syntax of the various message types

The semantics of the fields

Rules for determining when and how a process sends messages and responds to messages

Some application layer protocols are specified in RFCs - HTTP is RFC 7230.

These are public domain protocols.

Application-layer protocol is only one piece of a network application.

For example:

The Web is a client-server application that allows users to obtain documents from Web servers on demand. The Web application consists of many components, including a standard for document formats, Web browsers, servers, and an application-layer protocol, HTTP.

HTTP Defines the format and sequence of messages exchanged betwen browser and Web server. Thus, HTTP is only one piece fo the Web application.

The Web and HTTP

The World Wide Web arrived in the early 1990s. The Internet befoe that was mainly used by researchers and academics.

The Web operates on demand, unlike radio and television.

Overview of HTTP

Hypertext Transfer Protocol - is the heart of the web. Defined in RFC 1945, RFC 7230, and RFC 7540. It is implemented in two programs: a client and a server. These are executed on different end systems.

The client and the server talk to each other by exchanging HTTP messages.

Web Page/Document: consists of objects - HTML file, JPEG image, Javascript, CSS file, video, etc - that is addressable by a single URL. Most web pages consist of a base HTML file and several referenced objects.

Web Browsers - in the context of Web, they implement the client side

Web Servers - implement the server side of HTTP

HTTP utilizes TCP as the underlying transport protocol - Reliable data transfer. HTTP doesn’t worry about lost data or the details of TCP.

The HTTP client initiates a TCP connection with the server. Once the connection is established, the browser and the server processess access TCP through their socket interfaces.

On the client side the socket interface is the door between the client process and the TCP connection

On the server side it is the door between the server process and the TCP connection

HTTP servers retain no information about the clients - HTTP is a stateless protocol

Non-Persistent and Persistent Connections

  1. Non-Persistent Connections:
    • A new TCP connection is established for each request-response pair.
    • After the server sends the response, the connection is closed.
    • If the client needs to send another request, it must establish a new connection again.
    • This approach introduces extra overhead due to the repeated TCP connection setup and teardown, which can increase latency.
  1. Persistent Connections:
    • A single TCP connection is maintained for multiple request-response exchanges.
    • The connection stays open after a response is sent, allowing further requests to be sent over the same connection.
    • This reduces the overhead of repeatedly opening and closing connections, leading to better performance and lower latency.

Web Caching

A Web Cache/Proxy Server - a network entity that satisifed HTTP requests on behalf of an origin Web Server.

The browser establishes a TCP connection with the Web Cache and sends an HTTP request

The Web Cache checks to see if it has a copy of the object stored locally. If it does it sends it to the client

If it does not, then the Web Cache opens a TCP connection to the origin server.

When it receives the object, the web cache stores it locally and sends a copy within an HTTP response message to the client over the same TCP Connection.

A cache is both a server and client.

ISPs typically install Web caches. A university might install a cache on its campus network and configure all of the campus browsers to point to the cache.

Web Caching is used for faster load times - to reduce the response time for a client, particularly if the bottleneck bandwidth between client and origin server is much less than between client and web cache. It is used to reduce traffic on an institution’s access link to the internet.

The total response time for loading a web object is the sum of the LAN delay , access delay, and Internet delay.

Traffic intensity on the LAN is (15 requests/sec) * (1 MBits/request)/100 Mbps = 0.15

Traffic on the access link is (15 request/sec) * (1 Mbits/request)/(15 Mbps) = 1

LAN delay is negligible.

As the traffic intensity approaches 1, as in the access link, the delay becomes very large and grows without bound.

We can increase access speed from 15 Mbps to 100 Mbps - that is expensive.

Instead, installing a cache solves the issue. If the cache has a hit rate of 40 percent (0.4, typcially in the 0.2 to 0.7 range), 40 percent of the requests will be satisifed within 10 milliseconds. With only 60 percent of the requested objects passing through the access link, the traffic intensity on the link is reduced from 1.0 to 0.6.

The Conditional GET

A Way to allow a cache to verify that its objects are up to date.

HTTP/2

The first version since hTTP/1.1 in 1997. The primary goals for HTTP/2 are to reduce perceived latency by enabling request and response multiplexing over a single TCP connection provide request prioritization and serverp ush, and provide efficient compression of HTTP header fields.

HTTP/1.1 uses persistent TCP connections - 1 connection per page. Develoeprs of Web browsers discovered that sneding all the objects in a Web page over a single TCP connection has a Head of Line Blocking Problem.

if a web page has a large file at the top and smaller objects below it, everything is sent over the same connection. If the large file gets stuck, the smaller objects must wait behind it, delaying the page load. HTTP/1.1 browsers work around this by opening multiple parallel TCP connections.

HTTP/2 gets rid of parallel TCP connections (or reduces the number). Reduces the number of sockets that need to be opened and maintained at servers, but also allows TCP congestion control to operate as intended.

HTTP/2 Framing

The HTTP/2 method is to break each message into small frames and interleave the request and response messages on the same TCP connection.

Suppose a web page contains 1 large video clip and 8 smaller objects

The server will receive 9 concurrent requests from any browser that wants to see this page.

Suppose all frames are of fixed length, the video clip consists of 1000 frames, and each of the smaller objects consists two frames. With fram interleaving, after sending one frame from the video clip, the first frames of each of the small objects are sent. After send the second frame, the last frames of each of the small objects are sent. Thus, all of the smaller objects are ent after sending a total of 18 frames. If it weren’t used, the smaller objects would only be sent after sending 1016 frames.

Response Message Prioritization and Server Pushing

HTTP/2 prioritizes messages by assigning each request a priority weight (1 to 256). A higher number means higher priority. Using these weights, the server can send first the frames for the responses with the highest priority. The client also states each message’s dependency on other messages by specifying the ID of the message on which it depends.

HTTP/2 also allows a server to send multiple responses for a single client. In addition to the original request, the server can push additional objects to the client, without the client having to request each one.

HTTP/3

QUIC is a new transport protocol implemented over the bare-bones UDP protocol. There is message multiplexing, per-stream flow control, low latency connection establishment. HTTP/3 a new protocol that is designed to operate over QUIC. Many of HTTP/2’s features are used by QUIC.

Electronic Mail in the Internet

Modern email can include attachments, hyperlinks, HTML formatted text, and embedded photos.

Internet Mail System has three major components: User Agents, Mail Servers, and Simple Mail Transfer Protocol (SMTP).

User Agents are things like Outlook, Apple Mail, Web-based Gmail, Gmail App on smartphone. They allow users to read, reply to, forward, save, and compose messages.

Alice sending a message to Bob: When she is finished, her user agent sends the message to her mail server, where the message is placed in the outgoing message queue. When Bob wants to read a message, his user agent retrieves the message from his mailbox in his mail server.

Each recipient has a mailbox located in the mail server. If Alice’s mail sever cannot deliver to Bob’s mail server, Alice’s servers the message in a message queue and attempts to transfer the message later.

SMTP

SMTP transfers messages from senders’ mail servers to the recipients’ mail servers.

Originally restrited to 7-bit ASCII, as SMTP was before the time of large file transfers over mail (1982). Now, SMTP requires binary multimedia data to be encoded to ASCII before being sent over SMTP.

Alice invokes her user agent for e-mail, provides Bob’s e-mail address (for
example, bob@someschool.edu), composes a message, and instructs the
user agent to send the message.

Alice’s user agent sends the message to her mail server, where it is placed in a
message queue.

The client side of SMTP, running on Alice’s mail server, sees the message in the
message queue. It opens a TCP connection to an SMTP server, running on Bob’s
mail server.

After some initial SMTP handshaking, the SMTP client sends Alice’s message
into the TCP connection.

At Bob’s mail server, the server side of SMTP receives the message. Bob’s mail
server then places the message in Bob’s mailbox.

Bob invokes his user agent to read the message at his convenience.

SMTP does not use intermediary servers - if Alice’s is in Hong Kong and Bob’s is in St Louis, the TCP connection is a direct connection between the Hong Kong and St Louis.

The client SMTP establishes a TCP connection to port 25 at the server SMTP.When the connection is being established, the SMTP client indicates the email address of the sender and the email address of the recipient. Once the connection is established, the client sends a message.

Mail Message Formats

When an email message is sent, a header containing peripheral information precedes the body of the message itself. The header lines and the body of the message are separated by a blank line.

Every header must have a From: header line and a To: header line.

Mail Access Protocols

If a user agent is local, it is tempting to place the mail server locally as well. However, the problem with that is there does need to be an always on mail server exposed to the internet.

Alice is sending a message to Bob. Alice’s user agent does not dialogue directly with the recipient’s mail server. Instead the user agent sends the email to the sender’s mail server, and the mail server uses SMTP to send it to the recipient’s mail server.

Alice’s mail server can then try to send the message to Bob’s mail server (for example every 30 minutes) until the server accepts it.

Bob will use either HTTP (if he is using something web based like Gmail) to retrieve the email. Mail clients like Outlook use Internet Mail Access Protocol to retreive the email.

DNS - The Internet’s Directory Service

The identifier for a host is its hostname - www.facebook.com, google.com. Routers cannot processes these - they process IP addresses. 143.45.52.2 for example.

Services Provided by DNS

Domain Name System - people prefer hostnames over IP addresses. DNS is a directory service that translates hostnames to IP addresses.

DNS is a distributed database in a hierarchy of DNS servers and an application layer protocol that allows hosts to query the distributed database. They are often UNIX machines running the Berkeley Internet Name Domain software. DNS runs over UDP and uses port 53.

DNS is commonly employed by other protocols like HTTP and SMTP to translate user-supplied hostnames to IP address.

A client requests a URL. The users’s host must obtain the server for that URL.

The user machine runs the client side of the DNS application.

The browser extracts the hostname from the URL and passes the hostname to the client side of the DNS application.

The DNS client sends a query containing the hostname to a DNS server.

The DNS client eventually receives a reply, which includes the IP address for the hostname.

Once the browser receives the IP address from DNS, it can initiate a TCP connection to the HTTP server process located at port 80 at that IP address.

Other services:

Host Aliasing: A host with a complicated hostname can have an alias. For example, relay1[.]west-coast[.]enterprise[.]com can become enterprise[.]com relay1[.]west-coast[.]enterprise[.]com is the canonical hostname.

Mail Server Aliasing: DNS can be invoked by a mail application to obtain the canonical hostname for a supplied alias name.

Load Distribution: Busy sites like cnn[.]com are replicated over multiple servers, with each server running on a different end system and each having a different IP address.

Overview of How DNS Works

DNS is distributed by Design, since having a centralized database in 1 server wouldn’t scale.

There is a large number of servers organized in a hierarchical fashion and distributed around the world.

The hierarchy of DNS servers:

Root DNS servers: More than 1000 root servers that are copies of 13 different root servers, managed by 12 different organizations and coordinated through the IANA. ROot DNS servers provide the IP address of the TLD servers.

Top Level Domain Servers - top level domains such as com, org, net, edu, gov, country domains like uk, fr, ca, jp, there is a TLD server.

Authoritative DNS Servers - every organization with publicly accessible hosts (web servers and mail servers for example) on the Internet must provide publicly accessible DNS records that map the names of htose hosts to IP addresses.

Then there is the local DNS server - each ISP has a local DNS server. When a host connects to an ISP, the ISP provides the host with the IP addresses of one or more of its local DNS servers.

DNS Caching

DNS Caching - when a DNS server receives a DNS reply, it can cache the mapping in its local memory.

DNS Records and Messages

DNS Servers store resource records, including RRs that provide hostname-to-IP address mappings. Each DNS reply message carries one or more Resource Record.

A resource record is a four-tuple that contains the following fields:

(Name, Value, Type, TTL)

Meaning of Name and Value depend on Type

If Type=A, Name is a hostname and Value is the IP address for the hostname

If Type=NS, then name is a domain and Value is the hostname of an authoritative DNS server that knows how to obtain the IP addresses for hosts in the domain.

If Type=CNAME, then Value is a canonical hostname for the alias hostname Name

If Type=MX, then Value is the canonical name of a mail server that has an alias hostname Name

Peer to Peer File Distribution

In a P2P, pairs of intermittently connected hosts called peers communicate directly with each other. The peers are not owned by a service provider, but are instead PCs, laptops, and smartphones controlled by users.

Example: BitTorrent

The Web, email, DNS, all employ client-server architectures. In a P2P, there is no reliance on always-on infrastructure servers. Instead, pairs of intermittently connected hosts communicate directly with each other. These are PCs, laptops, and smartphone controlled by users.

Scalability of P2P Architectures

Servrs and peers are connected to the internet.

u_s = server’s upload rate

u_i = the upload rate of the ith peer

d_i = the download rate of the ith peer

F = the size of the file to be distributed in bits

N = the number of peers that want to obtain a copy of the file

Distribution time is the time it takes to get a copy of the file to all N peers.

Client-Server Architecture

The server must transmit one copy of the file to each of the N peers. Thus the server must transmit NF bits. The time to distribute the file must be at least NF/u_s.

d_min = download rate of the peer with the lowest download rate, the minimum of {d_1, d_p,…,d_n}. The peer wit the lowest download rate cannot obtain all F bits of the file in les than F/d_min seconds. The minimum distribution time is F/d_min

Thus the distribution time in a client-server architecture is the max(NF/u_s, F/d_min).

P2P Architecture

At the beginning of the distribution, only the server has the file. To get the file into the community of peers, the server must send each bit of the file at least once into its access link. Thus, the minimum distribution time is at least F/u_s.

The minimum distribution time is at least F/d_min.

The total upload capacity of the system as a whole is equal to the the upload rate of the server plus the upload rates of each of the inidividual peers. u_total = u_s + u_1 + u_2 + … + u_n.

The system must deliver F bits to N peers, thus deliver a total of NF bits that cannot be done faster than u_total. The minimum distribution time is also at least NF/u_total.

The minimum distribution time for P2P is the max(F/u_s, F/d_min, NF/u_total)

BitTorrent

The collection of all peers participating in the distribution of a particular file is called a torrent.

Peers in a torrent download equal-sized chunks of the file from one another, with a typical chunk size of 256 KBytes. When a peer first joins a torrent, it has no chunks. Over time, it accumulates more and more chunks. While it downloads chunks it also uploads chunks to other peers. Once a peer has the file it can leave the torrent or remain and continue to upload chunks to other peers.

Each torrent has an infrastructure node called a tracker. When a peer joins a torrent, it registers itself with the tracker. The tracker randomly selects a subset of peers from the set of participating peers, and sends the IP address to the new peer. That peer attempts to establish concurrent TCP connections with the ‘neighbors’.

As time goes on, some peers may leave and other peers may establish new connections, so a peer’s neighbors will fluctuate.

Periodically, a peer will ask neighbors for the list of the chunks they have. If a peer has L different neighbors, they will receive L list of chunks. With this knowledge, the peer will issue requests for chunks they do not have.

Rarest First is a technique used to determine the chunks that are the rarest among the neighbors and request those chunks first.

BitTorrent uses a trading algorithm that gives priority to the neighbors that are currently supplying her data at the highest rate, and then reciprocates by sending chunks to these same four peers.

Video Streaming and Content Distribution Networks

Straming video accounts for almost 80% of Internet Traffic in 2020.

Internet Video

In streaming stored video applications, the underlying medium is prerecorded video placed on servers and sent to users to view the videos on demand.

A video is a sequence of images displayed at a constant rate - 24 or 30 images per second.

From a networking perspective, the most salient characteristic of video is its high bit rate.

Compressed Internet video typically ranges from 100 kbps for low-quality Video to over 4 Mbps for streaming high-definition movies; 4K streaming envisions a bitrate of more than 10 Mbps. This can translate to a high amount of traffic and storage, particularly for high-end video. To provide continuous playout, the network must provide an average throughput to the streaming application that is at least as large as the bit rate of the compressed video.

HTTP Streaming and DASH

In HTTP streaming, the video is simply stored at an HTTP server as an ordinary file with a specific URL. This has a major shortcoming - all client receive the same encoding of the video, despite large variations in the amount of bandwidth available to a client.

Dynamic Adaptive Streaming over HTTP - DASH

The video is encoded into several different versions, with each version having a different bit rate and correspondingly, a different quality level. The client dynamically requests chunks of video segments of a few seconds in length.

DASH allows clients with different Internet access rates to stream in video at different encoding rates. Clients with low-speed connections can receive a low bit-rate version, and clients with high-speed connections can receive a high-quality version.

Content Distribution Networks

Many internet video companies are distributing on-demand multi-Mbps streams to millions of users on a daily basis. This is a challenging task.

One solution is to build a massive data center. This has some issues. If the client is far from the data center, packets will cross many communication links and likely pass through many ISPs. If one of these links provides a throughput that is less than the video consumption rate, the end-to-end throughput will also be below consumption rate, resulting in annoying freezing delays.

A popular video will likely be sent over the ame communication links - the Internet video company itself will be paying its provider ISP for sending the same bytes into the Internet over and over again.

A single data center represents a single point of failure.

Almost all major video-streaming companies make use of CDNs - manages servers in multiple geographically distributed locations, stores copies of the videos in its servers, and attempts to direct each user request to a CDN that will provide the best user experience.

CDNS typically adopt one of the two different server placement philosophies:

  1. Enter Deep: Deploying server clusters in acces ISPs all around the world.
  1. Bring Home - Building large clusters at smaller number of sites. Instead of getting into access ISPs, they instead place their clusters in IXPs.

CDN Operations

When a browser in a user’s host is instructed to retrieve a specific video, the CDN must intercept the request to determine a suitable CDN server cluster and redirect that client’s request to a server in that cluster. Most CDNs take advantage of DNS to intercept and redirect requests.

Cluster Selection Strategies

Cluster selection strategy - a mechanism for dynamically directing clients to a server cluster or a data center within the CDN. One strategy is to assign the client to the geographiclaly closest one.

Real-time measurements are used to provide the best cluster based on the current traffic conditions.
