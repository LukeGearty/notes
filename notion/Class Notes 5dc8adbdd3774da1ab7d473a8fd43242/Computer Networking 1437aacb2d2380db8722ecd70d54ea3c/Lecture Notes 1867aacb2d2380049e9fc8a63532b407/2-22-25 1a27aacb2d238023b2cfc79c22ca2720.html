Transport Services and Protocols

Provide logical communication between applicatin processes running on different hosts

Transport protocols in end systems:

sender: breaks application messages into segments, passes to network layer

Receiver: reassembles segments into messages, passes to application layer

Two transport protocols available to Internet applications: TCP and UDP

network layer: logical communication between hosts

transport layer: logical communication between processes

The transport layer takes the application message and adds its headers to create a segment, then sends it down to network layer

Two Principal Internet Transport Protocols:

TCP - reliable, in-order delivery with congestion control, flow control, and connection setup

UDP - unreliable, unordered delivery, with no-frills extension of “best-effort” IP

services that are not available are delay guarantees and bandwidth guarantees

Multiplexing at sender: handle data from multiple sockets, add transport layer headers

Demultiplexing at sender: uses header to send data to correct socket

Host receives IP datagrams, each datagram has source IP address, destination IP address

Each datagram carries ONLY ONE transport-layer segment

Each segment has source, destination port number

host uses IP addresses & port numbrs to direct segment to appropriate socket

Connectionless Demultiplexing

When creating socket, must specify host-local port #

When creating datagram to send into UDP socket, must specify: destination IP address and port number

When receiving host receives UDP segment it checks destination port #, and directs UDP segment to socket with that port #.

IP/UDP datagrams with same dest. port # but different source IP addresses and/or source port numbers will be directed to same socket at receiving host.

Sometimes for UDP the socket will run out of memory.

Connection-Oriented Demultiplexing

TCP socket is identified by a 4-tuple: source IP address and source port number, dest IP address and dest port number

The receiver uses all four value to direct segment to appropriate socket

Server may support many simultaneous TCP sockets: each socket identified by its own 4-tuple, each socket associated with a different connecting client

Different browsers will have different source ports (if you have Edge and Chrome going to the same site for example)

Different tabs will have different source ports

UDP Demultiplexes using destination port number (only)

TCP demultiplexes using 4-tuple, source and destination IP addresses and port numbers

In connectionless, there is only a single socket on the server handling all inbound traffic to that port. For connection-oriented, each inbound connection gets it's own dedicated socket on the server

Principles of Reliable Data Transfer

In an abstracted service, the channel is always reliable.

In the implementation, it is sending it over an unreliable channel - IP - it can lose, corrupt, or reorder data

Sender, receiver do not know the “state” of each other (was the message received)

Interfaces:

The sending process sends some kind of data, it creates a process (rdt_send()) and passes it over to the transport layer (udt_send()), which needs to reliably transport it. it sends it through an unreliable channel. When it arrives, the receiving host calls rdt_rcv() and uses deliver_data() to give it to the upper layer

rdt1.0

Reliable transfer over a reliable channel

Underlying channel is perfectly reliable - there are no bit errors and no loss of packets

rdt2.0

If the underlying channel flips bits in packet - checksum to detect bit errors

How to recover from errors?

The concepts of ACKS and NAKS - ACKS tells sender that the packet was received, the NAKS tell the receiver that the packet had errors

Sender retransmits packet on receipt of a NAK

The sender stops and waits for recieving an ACK or NAK

What happens if ACK/NAK is corrupted?

Sender adds sequence numbers to each packet - receiver discards duplicate packet

rdt2.1

Sender:

Seq # added to pkt

Two seq #s (0, 1) will suffice. Why?

Must check if received ACK/NAK corrupted

Twice as many states - state must “remember” whether expected pkt should have seq # of 0 or 1

Receiver:

Must check if received packet is duplicate

State indicates whether 0 or 1 is expected packet sequence #

Note: receiver can not know if its last ACK/NAK received OK at sender

rdt2.2

same functionality as rdt2.1, using ACKs only

Instead of NAK, receiver sends ACK for last packet received OK

Receiver must explicityl include seq # of pkt being ACKed

Duplicate ACK at sender results in same action as NAK: retransmits current pkt

rdt3.0

New channel assumptions: underlying channels can also lose packets

Sender waits a “reasonable” amount of time for ACK

retransmits if no ACK received in this time

If packet is just delayed: retransmission will be duplicate, but seq #s already handles this

receiver must specify seq # of packet being ACKed

Use countdown timer to interrupt after “reasonable” amount of time

The utilization is significantly slowed down by waiting for the next packet

Pipelining

Sender allows multiple, in-flight but not yet acknowledged packets

Range of sequence numbers must be increased

Buffering at sending and/or receiver

Go-Back-N

sender “window” of up to N, consecutive transmitted but unACKed packets

k-bit seq # in packet header

cumulative ACK - ACks all packets up to, including seq # n

On receiving ACK(n): move window forward to begin at n+1

timer for oldest in-flight packet

timeout (n): retransmit packet n and all higher seq # packets in window

*If the last ACK for the transmission is lost, the whole thing is sent again
