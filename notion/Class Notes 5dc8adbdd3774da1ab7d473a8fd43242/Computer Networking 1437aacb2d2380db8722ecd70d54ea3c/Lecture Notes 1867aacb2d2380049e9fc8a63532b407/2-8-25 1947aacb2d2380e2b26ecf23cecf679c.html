Creating a network app: the idea is that we can write programs that run on different end systems

For example, a Linux web server serving clients on Mac or Windows systems

We don’t have to do anything to ensure that these different end systems can run the same program - The core devices operate independently of the end systems. The highest level protocol for core devices is IP. They don’t run application level protocols like HTTP, DNS, SMTP, or even TCP/UDP.

Client-Server Paradigm

The Server is an always-on host, permanent IP address, often in data centers for scaling

The client contacts/communicates with server may be intermittently connected - may have dynamic IP address

Examples: HTTP, IMAP, FTP

Peer-Peer Architecture

No always-on server

Arbitrary end systems directly communicate

Peers request service from other peers, provide service in return to other peers

Self scalability - new peers bring new service capacity, as well as new service demands

Peers are intermittently connected and change IP addresses

Processes Communicating:

Process is an instance of a program running within a host

Within the same host, two processes communicate using inter-process communication (defined by OS)

Processes in different hosts communicate by exchanging messages

A socket is analoguous to a door

Process sends/receives messages to/from its socket

Sending process shoves message out door

Addressing processes

To receive messages, process must have an identifier

Host device has unique 32-bit IP address

Identifier includes both IP address and port numbers associated with process on host

Ex: HTTP over port 80, mail server over port 25

Multiple processes can run on the same host

Application-layer protocol defines:

types of messages exchanged: e.g., request, response

message syntax: what fields in messages & how fields are delineated

message semantics: meaning of information in fields

rules for when and how processes send & respond to messages

open protocols: defined in RFCs, everyone has access to protocol definition

What transport service does an app need?

Data integrity - some apps require 100% reliable data transfer, others can tolerate some loss

Timing - some apps (like internet telephony, interactive games) require low delay to be effective

throughput - some apps (e.g., multimedia) require minimum amount of throughput to be ‘effective’, others make use of what they get

Throughput is the actual amount of data successfully transmitted over a network in a given time, usually measured in bits per second (bps). It differs from bandwidth, which is the maximum theoretical data transfer rate, because real-world factors like congestion, latency, and packet loss can reduce throughput. Essentially, throughput represents the network’s real performance rather than its potential capacity.

security

Internet Transport Protocols Services:

TCP Service:

Reliable transport

flow control - sender won’t overwhelm receiver

congestion control: throttle sender when network overloaded

does not provide: timing, minimum throughput guarantee, security

connection-oriented: setup required between client and server processes

UDP service:

Unreliable data transfer: between sending and receiving process

Does not provide: reliability, flow control, congestion control, timing ,throughput guarantee, security, or connection setup

Web and HTTP

Web page consists of objects, each of which can be stored on different Web Servers

Web Page consists of base HTML-file which includes several referenced objects, each addressable by a URL

Client/Server model: client is a browser that requests, receives and displays Web objects

Server is a web server that sends (using the HTTP protocol) objects in response to requests

HTTP (unless HTTP 3) uses TCP - client initiates TCP connection to server, port 80 or port 443 for HTTPS

Server accepts TCP connection from client

HTTP messages exchanged between browser (HTTP client) and Web Server (HTTP server)

TCP connection closed

HTTP is stateless - server maintains no information about past client requests

Non-persistent HTTP - the TCP connection is opened, at most one object sent over TCP connection, then TCP connection closed

Downloading multiple objects required multiple connections

Persistent HTTP - TCP connection opened to a server, multiple objects can be sent over single TCP connection between client, and that server, then TCP connection closed

If there is a network with an unreliable performance/connection, then non-persistent may make more sense

Non-persistent HTTP

RTT - time for a small packet to travel from client to server and back - round trip time

HTTP response time (per object):

One RTT to initiate TCP connection

One TT for HTTP request and first few bytes of HTTP response to return

object/file transmission time

Non-persistent HTTP response time = 2RTT + file transmission time

HTTP request message

Two types of HTTP messages: request, response

\r\n - carriage return character, line-feed character

POST method: web pages often includes form input, user input sent from client to server in entity body of HTTP post request message

GET method (for sending data to server): include user data in URL field of HTTP GET request message (following a ‘?’) - SQL injection is done through a GET method

HEAD method - like a GET method except for headers

PUT method - updating something on the server

Maintaining user/server state: cookies

HTTP GET/response interaction in stateless - no information is retained about the users

Cookies are used to maintain some state between transactions

Four components of cookies:

Cookie header line of HTTP response message

Cookie header line in next HTTP request message

Cookie file kept on user’s host, managed by user’s browser

back-end database at Web site

Web Caches (proxy servers)

Goal: satisfy client request without involving origin server

User configures browser to point to a Web Cache

Browser sends all HTTP requests to cache

if object in cache: cache returns object to client

else cache requests object from origin server

Conditional GET:

Goal: don’t send object if cache has an up-to-date version

If not up to date, get the object from the origin server

HTTP/2:

Key goal: decreased delay in multi-object HTTP requests

HTTP/3: key goal is to decrease delay in multi-object HTTP requests
