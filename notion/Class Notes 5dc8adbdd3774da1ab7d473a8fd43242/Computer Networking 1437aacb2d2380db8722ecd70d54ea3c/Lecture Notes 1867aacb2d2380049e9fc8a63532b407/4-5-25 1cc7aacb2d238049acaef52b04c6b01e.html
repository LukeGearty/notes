Network-layer functions

forwarding: move packets from router’s input to appropriate router output - this is on the data plane, implemented in hardware

routing: determine route taken by packets from source to destination - this is on the control plane, implemented in software. It is fast, but not as fast as the hardware.

Two approaches to structuring network control plane:

per router control (traditional)

logically centralized control (software defined networking)

In the per-router control, there are individual routing algorithm components in each and every router interact in the control plane

In the SDN, remote controller computes, installs forwarding tables in routers

Routing Protocols

The goal is to determine a “good” path from sending hosts to receiving host, through network of rotuers

The path is a sequence of routers packets traverse from given initial source host to final destination host

“good” means least “cost”, “fastest”, “least congested”

Graph Abstraction

Graph: G = (N, E)

N is a set of routers

E is the set of links between routers

C(a, b) is the cost of direct links between a and b

Cost is defined by network operator: could always be 1, or inversely related to bandwidth or congestion

Routing algorithm classification

Global: all routers have complete topology, link cost info. A complete view of the entire network. These are called “link state” algorithms. Good in small networks, not so good in big networks.

Decentralized: iterative process of computation, exchange of info with neighbors. Routers initially only know link costs to attached neighbors. These are called Distance Vector algorithms.

Static: routes change slowly over time.

Dynamic: rotues change more quickly, periodic updates or in response to link cost changes.

Link State

Dijkstra’s Link-State Routing Algorithm

Centralized: network topology, link costs known to all nodes

accomplished via “link state broadcast”

all nodes have same info

computes least cost path from one node (source) to all other nodes

gives forwarding table for that node

iterative: after k iterations, know least cost path to k destinations

Notation:

C(x,y) direct link cost from node x to y; =∞ if no direct

D(v) current estimate of cost of least-cost path from source to destination v

P(v) predecessor node along path from source to v

N’: set of nodes whose least-cost-path definitively known

N: set of the network nodes

N’ is only set to be the source. N’ = {u}

go through the nodes, v, connected to the source, set D(v) to be C(u, v). else D(v) = ∞

Then, find a node, w, such that D(w) is a minimum.

Add w to N’, and update D(v) for all neighbors v of w. D(v) will be the minimum of either the old value of D(v), or D(w) + c(w,v).

This ends when N’ == N.

algorithm complexity: n nodes

each of n iterations, need to check all nodes, w, not in N

n(n +1) / 2 comparisons: O(n^2)

more efficient implementations possible O (n log n)

Oscillations possible

Oscillations means you can have false routing methodology

When Link costs depend on traffic volume, route oscillations possible

Distance Vector Algorithm

Based on Bellman-Ford (BF) equation (dynamic programming):

Let D_x(y) vbe the cost of least-cost path from x to y

Then D_x(y) = min { c(x,y) + D_v(y) }

D_x(y) is the minimum cost of x to y + the cost of node v to y

Each node: waits for change in local link cost or msg from neighbor → recompute DV estimates using DV received from neighbor → if DV to any destination changes, notify neighbor

Iterative, asynchronous: each local iteration caused by: local link cost change, DV update message from neighbor

Distributed, self-stopping: each node notifies neighbors only when its DV changes

Neighbors then notify their neighbors, only if necessary

No notification received, no actions taken
