Chapter Two

Principles of the Application Layer

The Application Layer is where it all starts. This is what the user sees.

Some Network Apps that the users use: social networking, web, text messaging, e-mail, multi-user network games, streaming, P2P, voice over IP, real-time video conferencing, Internet Search, remote login… etc

The designers did not design for the Internet to specifically support all of these different applications, but it does.

Creating a Network app, you must write programs that run on different end systems, communicate over network.

Web Server software communicating with browser software is an example of this.

There are two paradigms that describe how the pieces of the network app interact with each other.

The Client-Server Model

There is a server - always on with a permanent IP address that can be discovered by clients.

Client - contact, communicate with the server - may be intermittently connected, dynamic IP addresses, do not communicate with each other

HTTP, IMAP, FTP

Peer 2 Peer

No always on server - arbitrary end systems directly communicate

Peers request service from other peers, provide service in return to other peers.

Self scalability - new peers bring new service capability and demands

File sharing

Process Communicating - a process is a program running within a host. An instance of a running program.

WIthin the same host, two processes communicate using inter-process communication defined by the OS

Processes in different hosts communicate by exchanging messages

The client is the process that starts the communication

The process sends/receives messages to/from its socket, like a door for the process - sending process shoves message out door

Sending process relies on transport infrastructure on other side of door to deliver message to socket at receiving process

Two sockets are involved on both sides

Addressing processes - to receive messages, process must have identifier. The host device has unique 32-bit IP address.

Identifier includes both IP address and port numbers associated with process on host.

HTTP - port 80

mail server - port 25

An application-layer protocol defines the types of messages exchanged (request and response), message syntax (What fields in messages and how fields), message semantics (meaning of information in the message), and rules for when and how processes send and respond to messages.

Transport services:

  1. Data integrity - some apps require 100% reliable data transfer, other apps can tolerate some loss
  1. Timing - some apps require low delay to be effective - like games or telephony
  1. Throughput - some apps require minimum amount of throughput to be effective, other apps are more elastic and can use whatever throughput they can get
  1. Security - encryption, data integrity

Internet Transport Protocols Service

TCP - reliable transport, flow control (won’t overwhelm receiver), congestion control won’t throttle network, connection-oriented (requires setup).

UDP - unreliable data transfer (makes best effort), does not provide reliability, flow control, congestion control, timing, throughput guarantee, security, or connection setup

Vanilla TCP & UDP sockets do not provide encryption, cleartext passwords sent into socket traverse Internet in cleartext.

The Web and HTTP

A Web page consists of objects - a base HTML file and a set of referenced objects like HTML files, JPEG image, audio file, etc. Referenced objects addressable by a URL.

HTTP - hypertext transfer protocol - utilizes the client-server model

Web’s application layer protocol

Utilizes TCP:

client initiates TCP connection to server on port 80

Server accepts TCP connection from client

HTTP messages (application-layer protocol messages) exchanged between browser (HTTP client) and Web Server (HTTP server)

Then TCP connection is closed

HTTP is stateless - server maintains no information about past client requests

There are two types of HTTP connections:

  1. Non-Persistent HTTP
    1. TCP Connection is opened - at most one object is sent - tcp connection is closed
    1. Downloading multiple objects required multiple connections
  1. Persistent HTTP
    1. TCP connection is opened to a server - multiple objects can be sent over single TCP connection between client and that server
    1. TCP connection is then closed

Non-persistent HTTP: response time

RTT - time for a small packet to travel from client to server and back

Non-persistent HTTP requires 2 RTTs - 1 to initiate the connection, a second to request the objects.

Non persistent HTTP response time = 2RTT + file transmission time

Non-persistent HTTP issues:

  1. requires 2 RTTs
  1. OS overhead for each TCP connection
  1. browsers often open multiple parallel TCP connections to fetch referenced objects in parallel

Persistent HTTP:

  1. Server leaves connection open after sending response.
  1. Subsequent HTTP messages between same client/server sent over open connection
  1. Client sends requests as soon as it encounters a referenced object
  1. As little as one RTT for all the referenced objects cuts response time in half to 1 RTT

Two types of HTTP messages

HTTP Requests:

Starts with a single request line (GET /index.html HTTP/1.1 |r|h)

General format is request line, header lines, and then body

POST Method - user sends information to server

GET method - user requests a page from server

HEAD method - like GET, except just the headers

PUT method - user uploads new object to server

HTTP Response:

Begins with status line - HTTP version and status code phrase (200 OK being an example)

Cookies are used to maintain user/server state. They maintain some state between transactions.

Four components:

  1. Cookie header line of HTTP response message
  1. Cookie header line in next HTTP request message
  1. Cookie file kept on user’s host, managed by user’s browser
  1. back end database

Cookies can be used for authorization, shopping carts, recommendations, user session state

How to keep state?

at protocol endpoints: maintain state at sender/receiver over multiple transactions

in messages: cookies in HTTP messages carry states

Web Caching: The goal is to satisfy client requests without involving the origin server

The user configures browser to point to a (local) web cache

browser sends all HTTP requests to cache

if the object is in the cache, the cache returns objects to clients

Else cache requests object from origin server, caches received object, then returns object to clients

The Web Cache acts in the client and server role

Server tells cache about objects allowable caching in response header

Why web caching: reduce response time for client request - the cache is closer to the client. It reduces traffic on an institution’s access link.

Scenario:

Access link rate: 1.54 Mbps

RTT from institutional router to server: 2 sec

Web object size: 100K bits

average request rate from browsers to origin server: 15/sec

avg data rate to browser: 1.50 Mbps

Avg data rate to browser is very close to the access link rate.

Access link utilization = .97 (97%)

This would cause a large queuing delay as the utilization is really high.

LAN Utilizatio: .0015

End-end delay = Internet delay + access link delay + LAN delay = 2 sec + minutes + usecs (microseconds)

To improve performance, there are a few options.

  1. Increase access link rate to 154 Mbps. This solves the problem, but this is very expensive to do.
  1. Install a web cache - web caches are cheaper, access link utilization will go down, average end-end delay will be better

Assume 40 % of requests are served by the web cache (cache hit rate is 0.4)

60 % of requests go to the internet.

0.6 * 1.50 Mbps = .9 Mbps.

Access link utilization goes down to 0.9/1.54 = 0.58 (58%), much lower than the 97% above.

End-end delay = 0.6 * (delay from origin servers) + 0.4 * )delay when satisifed at cache) = 0.6 (2.01) + 0.4 (~msecs) = ~1.2 seconds.

Conditional GET - goal is not to send object if cache has up-to-date cached

No object transmission delay (or use of network resources)

Client: specify date of cached copy in HTTP Request

if modified since

Server: response contains no object

HTTP/2 - the current version of HTTP. The key goal of HTTP/2 is to decrease delay in multi-object HTTP requests.

Increased flexibility at server in sending objects to client:

methods, status codes, most header fields unchanged from HTTP 1.1

transmission order of requested objects based on client-specified object priority

push unrequested objects to client

divide objects into frames, schedule frames to mitigate HOL blocking

Head of the Line Blocking:

HTTP 1.1 requests 1 large object (video file, and 3 smaller objects)

The bigger object will need to go through before the 3 smaller objects.

Like someone with a full cart of groceries in front of you at the store

HTTP/2 - object is divided into frames, fram transmission interleaved.

HTTP/2 to HTTP/3

HTTP/2 over single TCP connection means:

recovery from packet loss still stalls all object transmissions

as in HTTP 1.1, browser have incentive to open multiple parallel TCP connections to reduce stalling, increase overall throughput

no security over vanilla TCP connection

HTTP/3: adds security, per object error- and congestion-control (more pipelining) over UDP.

more on HTTP/3 in transport layer

DNS

Responsible for translating hostnames (like cnn.com) to IP addresses. It is a distributed application.

Internet Hosts have IP addresses and “names” to identify them. How do we map between IP addresses and names?

DNS - distributed database implemented in hierarchy of many name servers.

Application-layer protocols: hosts, DNS servers communicate to resolve names

Core internet function, implemented as application-layer protocol

DNS Services:

Hostname-to-IP address translation

host aliasing: canonical, alias names

Mail server aliasing

Load distribution: replicated Web servers, many IP addresses correspond to one name

A centralized DNS:

Would provide a single point of failure

High Traffic volume

Doesn’t scale

High maintenance

DNS is a highly distributed, high scale, high-performance database with billions of simple records.

There are trillions of queries per day.

Many more reads than writes

Performance matters: almost every internet transaction interacts with DNS

Organizationally, physically decentralized: millions of different organizations responsible for their records

Root DNS Servers

Then .com DNS, .org DNS, .edu DNS, Top Level Domain

Then yahoo.com DNS servers, nyu.edu, etc. These are authoritative

Client wants IP address for www.amazon.com

client queries root server to find .com DNS server

client then queries .com DNS server to get amazon.com DNS server

client then queries amazon.com DNS server to get IP address

DNS: root name servers - official, contact-of-last resort by name servers that can not resolve name

Security is very important

ICANN is responsible for root DNS domain. 13 logical root servers, with many physical srver replicated.

Top-Level Domain - TLD: .com, .org, .net, .edu, .aero, .jobs, country domains like .cn, .uk

Authoritative: organization’s own DNS server(s), providing authoritative hostname to IP mappings

Local DNS server: When a host makes DNS query, it is sent to its local DNS server

Iterated Query: The requested host sends the DNS query to local DNS server. The local DNS server queries the root DNS server, which returns the correct TLD server. The local DNS server queries the TLD server, which The TLD server responds with the authoritative server. The local DNS server then queries the authoritative DNS server, which replies with the IP address. The local DNS server then sends the IP address to the host.

Recursive Query: The requested host sends the DNS query to local DNS server. The local DNS server queries the root server, which queries the TLD server, which queries the authoritative DNS server, which replies with the IP address to the TLD server, which replies to the root server, which replies to the local DNS server.

Caching DNS Information - once any name server learns mapping, it caches mapping and immediately returns a cached mapping in response to a query.

Caching improves response time - cache entires timeout after some TTL

TLD servers typically cached in local name servers.

Cached entires may be out-of-date - if named host changes IP address, may not be known internet-wide until all TTLs expire. Best effort name-to-address translation.

DNS database records have four fields: name, value, type, TTL

Type=A: name is hostname, value is IP address

type=NS: name is domain, value is hostname of the authroitative name server for this domain

type=CNAME: name is alias for some “canonical” (the real) name, value is canonical name.

ex: www.ibm.com is really servereast.backup2.ibm.com

type=MX: value is name of SMTP server associate with name

Protocol Messages:

Message header:

identification: 16 bit number

flag: whether query or reply

recursion desired, available

reply is authoritative

questions (variable # of questions)

Resource Records in response to query

Video Streaming and Content Distribution Networks

Streaming video traffic is a major consumer of internet bandwidth - Netflix, YouTube, Amazon Prime.

There is the challenge of scale - how to reach ~1B users. A challenge of heterogeneity - different users have different capabilities.

Video is a sequence of images displayed at constant rate - e.g 24 images/sec

coding: use redundancy within and between images to decrease # bits used to encode image.

Spatial: instead of sending N values of same color, send only two value: color value and number of repeated values (N)

temporal coding: insteado f sending complete frame at i + 1, send only differences from frame i

Constant Bit Rate: video encoding rate fixed

Variable Bit rate: video encoding rate changes as amount of spatial, temporal coding changes

Main Challenges:

server-to-client bandwidht will vary over time, with changing network congestion levels (in house, access network, network core, video server)

packet loss, delay due to congestion will delay playout, or result in poor video quality
